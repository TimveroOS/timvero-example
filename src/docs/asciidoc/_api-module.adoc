[[api-module]]
= API Module — web configuration, error handling & OpenAPI
:sourcedir: ../../main/java/com/timvero/example/portal

The `com.timvero.api` module is the SDK's REST API foundation. When you add it to your `@ComponentScan`, you get:

* **Web MVC configuration** — converters for `MonetaryAmount`, `CurrencyUnit`, `CustomEnum`, `@Phone` fields, and Jackson modules for JSON serialization
* **Exception handling** — automatic translation of validation errors, binding failures, and bad request formats into structured JSON responses
* **OpenAPI documentation** — Swagger type mappings for monetary types, custom enums, and phone numbers

The module is designed to work alongside your own security and controller configuration — it handles the plumbing so you can focus on business endpoints.

[NOTE]
====
`ApiWebConfig` extends `DelegatingWebMvcConfiguration`, which takes over Spring MVC configuration. Do not declare another `@EnableWebMvc` or `WebMvcConfigurer` bean in the same application context — they will conflict.

== Web Configuration (ApiWebConfig)

`ApiWebConfig` is the central configuration class. It registers converters, formatters, Jackson modules, and the exception handler.

=== What It Configures

[cols="2,3", options="header"]
|===
|Area |Detail

|**Argument Resolvers**
|`UserWebArgumentResolver` — resolves the current user in controller method parameters

|**Formatters**
|`String` -> `CurrencyUnit` (via `MonetaryUtil::getCurrency`) +
`String` -> `MonetaryAmount` (via `MonetaryUtil::parse`) +
`CustomEnumConverter` for SDK custom enums +
`PhoneFormatterFactory` for `@Phone`-annotated fields

|**Jackson Modules**
|`Jdk8Module` (Optional support) +
`JavaTimeModule` (java.time serialization) +
`MonetaMoneyModule` (MonetaryAmount as `{"amount": 99.96, "currency": "USD"}`) +
`PhoneFieldDeserializer` (phone normalization on input)

|**Jackson Settings**
|`WRITE_BIGDECIMAL_AS_PLAIN = true` (no scientific notation) +
`WRITE_DATES_AS_TIMESTAMPS = false` (ISO-8601 strings)

|**Beans**
|`ApiExceptionControllerAdvice` — global error handler
|===

=== Phone Region

The phone formatter reads the region from `application.phone.region`:

[source,properties]
----
application.phone.region=GB
----

This affects both formatting (display) and deserialization (input parsing). All `@Phone`-annotated fields in your request DTOs will be normalized to international format using this region.

== API Documentation (ApiDocConfig)

`ApiDocConfig` activates only when SpringDoc is on the classpath (`@ConditionalOnClass`). It customizes the OpenAPI schema for SDK types.

=== Automatic Type Mappings

[cols="2,2,2", options="header"]
|===
|Java Type |OpenAPI Schema |Example Value

|`MonetaryAmount`
|Object with `amount` (BigDecimal) + `currency` (String)
|`{"amount": 99.96, "currency": "USD"}`

|`CurrencyUnit`
|String
|`"USD"`

|`@Phone String`
|String with pattern `^\+?[1-9]\d{7,14}$`
|`"+48123456789"`
|===

=== Registering Custom Enums

If your project uses `CustomEnum` types, register them so OpenAPI shows the allowed values:

[source,java]
----
@PostConstruct
protected void initConfig() {
    super.initConfig();
    ApiDocConfig.registerClass(Country.class);
    ApiDocConfig.registerClass(Employment.class);
}
----

Call `ApiDocConfig.registerClass()` from a `@PostConstruct` method in your own `@Configuration` class that extends `ApiDocConfig`, or from any `@PostConstruct` that runs after `ApiDocConfig` is initialized.

== Error Handling (ApiExceptionControllerAdvice)

The module registers `ApiExceptionControllerAdvice` at `@Order(950_000)`. It converts common exceptions into JSON responses with field-level error details.

=== Exception → Response Mapping

[cols="2,1,3", options="header"]
|===
|Exception |HTTP Status |Response Body

|`HttpMessageNotReadableException` (invalid JSON format)
|400
|`{"fieldName": "Invalid format: <value>"}`

|`BindException` (validation failure)
|400
|`{"field1": ["error1", "error2"], "field2": ["error3"]}`

|`ValidationException` (SDK checker rejection)
|400
|`{"field1": ["rejection message"], ...}`

|`MissingServletRequestParameterException`
|400
|`{"paramName": "Required field"}`
|===

Error messages are resolved from the request locale via `RequestMessages` — so they respect your `messages.properties` files.

=== Controller Parameter Resolvers

The module registers three `HandlerMethodArgumentResolver` beans that inject request-scoped values directly into controller parameters. No boilerplate to read headers or `HttpServletRequest`.

[cols="2,2,3", options="header"]
|===
|Annotation / Type |Source |Use

|`@IpAddress String`
|`HttpServletRequest.getRemoteAddr()`
|Audit logs, rate limiting, fraud detection — the client's IP as Spring sees it

|`@BrowserAgent String`
|`User-Agent` HTTP header
|Audit logs, device fingerprinting, analytics

|`UserAccount` / `BackOfficeUserDetails` (no annotation)
|`SecurityContext` principal resolved from the authenticated request
|Any controller that needs the current user — type-based resolution, not annotation-based
|===

Usage:

[source,java]
----
@PostMapping("/login-event")
public void recordLogin(
        @IpAddress String clientIp,             // <1>
        @BrowserAgent String userAgent,         // <2>
        UserAccount currentUser                 // <3>
) {
    auditLog.recordLogin(currentUser, clientIp, userAgent);
}
----
<1> From `com.timvero.common.web.IpAddress` annotation — parameter type must be `String`. Returned value is `request.getRemoteAddr()` — respects Spring's `ForwardedHeaderFilter` if configured, so behind a proxy you get the real client IP via `X-Forwarded-For`.
<2> From `com.timvero.common.web.BrowserAgent` annotation — same pattern, reads the `User-Agent` header.
<3> No annotation needed — `UserWebArgumentResolver` matches by type. Also accepts `BackOfficeUserDetails` if you need the full Spring Security principal. Returns `null` if no user is authenticated.

`UserWebArgumentResolver` is already registered by `ApiWebConfig` automatically. The `@IpAddress` and `@BrowserAgent` resolvers are provided by `com.timvero.common.web.IpAddressArgumentResolver` / `BrowserAgentArgumentResolver` — register them in your `WebMvcConfigurer` if the API module doesn't wire them for your deployment:

[source,java]
----
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {
    @Override
    public void addArgumentResolvers(List<HandlerMethodArgumentResolver> resolvers) {
        resolvers.add(new IpAddressArgumentResolver());
        resolvers.add(new BrowserAgentArgumentResolver());
    }
}
----

[NOTE]
====
**Behind a reverse proxy**: `request.getRemoteAddr()` returns the proxy's IP unless Spring's `ForwardedHeaderFilter` is active (e.g., `server.forward-headers-strategy=framework` in `application.properties`). Configure this before trusting `@IpAddress` for audit or security decisions.
====

== Best Practices

[.unstyled]
* {ok} **Use `@ComponentScan(basePackageClasses = {ApiWebConfig.class, ...})`** to wire the module — not manual `@Import`
* {ok} **Let `ApiExceptionControllerAdvice` handle validation errors** — don't catch `BindException` in controllers
* {ok} **Register `CustomEnum` types with `ApiDocConfig.registerClass()`** so Swagger shows allowed values
* {ok} **Set `application.phone.region`** to match your deployment region — affects all `@Phone` formatting
* {ok} **Use `@SecurityRequirement` on controllers** — documents auth in Swagger and enforces it consistently
* {ok} **Create project-specific exception handlers with `@Order(HIGHEST_PRECEDENCE)`** — they run before the SDK handler

== Anti-Patterns

[.unstyled]
* {bad} **Adding `@EnableWebMvc` or another `WebMvcConfigurer`** when `ApiWebConfig` is scanned — causes configuration conflict because `ApiWebConfig` extends `DelegatingWebMvcConfiguration`
* {bad} **Registering Jackson modules manually** when the module already does it — leads to duplicate converters and unpredictable serialization
* {bad} **Catching `BindException` in controller methods** — the global handler already produces structured JSON errors; catching it yourself loses the field-level grouping
* {bad} **Hardcoding phone patterns in OpenAPI `@Schema` annotations** — `ApiDocConfig` does this automatically for `@Phone` fields
* {bad} **Using `@Order` values above 950,000 for exception handlers** — they will load after the SDK handler and never fire for already-handled exceptions

== Production Checklist

[.unstyled]
* {todo} `application.phone.region` is set to the correct ISO country code
* {todo} `internal.api.username` and `internal.api.password` are overridden from defaults
* {todo} Swagger UI access is either secured or disabled in production (`springdoc.swagger-ui.enabled=false`)
* {todo} Custom `@RestControllerAdvice` handlers use `@Order(HIGHEST_PRECEDENCE)` or a value below 950,000
* {todo} All `CustomEnum` types used in request/response DTOs are registered via `ApiDocConfig.registerClass()`
* {todo} `MonetaryAmount` fields serialize as `{"amount": ..., "currency": "..."}` — verify in Swagger "Try it out"
* {todo} Validation error responses contain field-level messages, not raw exception text
