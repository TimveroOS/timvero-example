= Offer Engine & Credit Products
:sourcedir: ../../main/java

**Purpose**: Generate personalized loan offers based on credit products, participant data, and business rules.

The Offer Engine transforms static credit product templates into personalized offers by evaluating participant risk data, workflow results, and business rules through configurable scripts.

== Process Flow

The offer generation follows this sequence:

[cols="1,1,1,1,1", options="header"]
|===
|Product Template |Risk Assessment |Personalized Offer |Select Condition |Active Credit

|• Amount ranges
• Interest rates  
• Terms
• Payment engine
|• Credit score
• Income data
• Workflow results
• External data
|• Adjusted amounts
• Final rates
• Secured options
|• Participant chooses offer
• Payment schedule
• Signs contract
|• Credit entity
• Active servicing
|===

== What You'll Build

* **Credit Products**: Loan product templates with terms and parameters
* **Product Additives**: Specific configurations within products (rates, conditions)
* **Offer Generation**: Automated personalized offer creation
* **Secured Offers**: Collateral-based offer variants
* **Integration**: Workflow-driven offer generation

        // and remaining balances after payment

----

**Engine Purpose:**

* **Payment Breakdown**: Calculates how much of each payment goes to principal vs interest
* **Schedule Generation**: Creates complete payment timeline with remaining balances
* **Algorithm Flexibility**: Different engines can implement various payment calculation methods (annuity, differentiated, interest-only, etc.)

**Integration:**

* **Product Configuration**: Credit product's `engineName` field matches Spring service name
* **Automatic Discovery**: Platform automatically finds and manages all ScheduledEngine beans
* **Validation**: System ensures payment calculations are mathematically correct (remaining debt = 0)

== Secured Offers & Procuring Engines

Secured offers extend basic offers with collateral or guarantee requirements through the ProcuringEngine pattern.

=== Procuring Engine Pattern

The `ProcuringEngine` transforms basic product offers into secured variants with specific collateral requirements:

[source,java]
----
@Component
public class PenaltyProcuringEngine implements ProcuringEngine {

    @Override
    public ProcuringType procuringType() {
        return PENALTY;
    }

    @Override
    public Collection<? extends SecuredOffer> generateSecuredOffers(ProductOffer productOffer) {
        return List.of(new PenaltySecuredOffer((ExampleProductOffer) productOffer));
    }
}
----

**Engine Responsibilities:**

* **Procuring Type**: Defines what type of collateral/guarantee required
* **Offer Generation**: Creates secured variants from basic offers
* **Business Logic**: Implements specific procuring strategies
* **Flexibility**: Multiple engines can handle different collateral types

=== Procuring Types

Define the types of collateral or guarantee requirements:

[source,java]
----
@Configuration
public class ExampleProcuringType {

    public static final String CODE_PENALTY = "PENALTY";
    public static final ProcuringType PENALTY = new ProcuringType(CODE_PENALTY);
    
    // Other procuring types:
    // VEHICLE_COLLATERAL, PROPERTY_COLLATERAL, COSIGNER, etc.
}
----

**Common Procuring Types:**

* **NO_PROCURING**: Unsecured loans based on creditworthiness
* **PENALTY**: Higher rates with penalty clauses for default
* **VEHICLE_COLLATERAL**: Car loans with vehicle as collateral
* **PROPERTY_COLLATERAL**: Mortgages with property as collateral
* **COSIGNER**: Loans requiring guarantor/co-signer

=== Procuring Engine Integration

Procuring engines integrate with the main offer generation flow:

**Generation Process:**

1. **Basic Offers**: OfferEngine generates standard ProductOffers
2. **Procuring Analysis**: System identifies applicable procuring types
3. **Secured Generation**: Each ProcuringEngine creates secured variants
4. **Offer Portfolio**: Participant receives both basic and secured options
5. **Selection**: Participant chooses preferred offer type

**Business Benefits:**

* **Risk Mitigation**: Collateral reduces lender risk
* **Rate Optimization**: Secured offers can have lower interest rates
* **Market Expansion**: Serve customers who need collateral-based options
* **Regulatory Compliance**: Meet requirements for different loan types

== Integration Patterns

=== Offer to Credit Conversion

The conversion from offer to active credit happens in two phases: **condition selection** and **contract signature**.

**Phase 1: Offer Selection & Condition Creation**

When a participant selects an offer, the system creates a credit condition and prepares for contract signature:

[source,java]
----
@Controller
@RequestMapping("/submit-regular")
public class SelectRegularConditionAction extends SelectConditionAction<ExampleProductOffer, ConditionForm> {

    @Override
    protected EntityAction<? super ExampleProductOffer, ConditionForm> action() {
        return when(o -> o.getApplication().getCondition() == null
            && o.getApplication().getStatus().equals(ApplicationStatus.CONDITION_CHOOSING)
            && o.getParticipant().getStatus() == ParticipantStatus.APPROVED).then((offer, form, user) -> {
                
                Application application = offer.getApplication();
                ExampleSecuredOffer securedOffer = findSecuredOffer(offer, form.getSecuredOfferKey());

                // Calculate payment terms
                MonetaryAmount principal = form.getPrincipal();
                BigDecimal interestRate = offer.getProductAdditive().getInterestRate();
                Integer term = form.getTerm();
                
                MonetaryAmount regularPayment = PaymentCalculator.calcAnnuityPayment(
                    principal, MonetaryUtil.zero(principal.getCurrency()), 
                    periodicInterest(Period.ofMonths(1), interestRate), term, 0);

                // Create credit condition
                ExampleCreditCondition condition = new ExampleCreditCondition(
                    principal, offer.getCreditProduct().getEngineName(), 
                    interestRate, offer.getCreditProduct().getLateFeeRate(),
                    Method_30_360_BB.NAME, Period.ofMonths(1), term, 
                    regularPayment, securedOffer);
                
                application.setCondition(condition);

                // Generate payment schedule
                PaymentSchedule paymentSchedule = scheduledService.getPaymentSchedule(
                    condition, form.getPrincipal(), form.getStart());
                application.setPaymentSchedule(paymentSchedule);

                // Move to contract signature phase
                application.setStatus(ApplicationStatus.PENDING_CONTRACT_SIGNATURE);
                
                // Generate contract document
                documentService.generate(application.getBorrowerParticipant(),
                    ParticipantDocumentTypesConfiguration.APPLICATION_CONTRACT,
                    offer.getCreditProduct().getUuidContractTemplate());
            });
    }
}
----

**Phase 2: Contract Signature & Credit Creation**

When the contract is signed, an EntityChecker automatically creates the active credit:

[source,java]
----
@Component
public class ContractSignChecker extends EntityChecker<Application, UUID> {

    @Override
    protected void registerListeners(CheckerListenerRegistry<Application> registry) {
        registry.entityChange(SignableDocument.class,
                d -> participantRepository.getReferenceById(d.getOwnerId()).getApplication())
            .updated(SignableDocument_.STATUS)
            .and(d -> d.getStatus() == SignatureStatus.SIGNED
                && d.getDocumentType() == ParticipantDocumentTypesConfiguration.APPLICATION_CONTRACT);
    }

    @Override
    protected boolean isAvailable(Application application) {
        return application.getStatus().equals(ApplicationStatus.PENDING_CONTRACT_SIGNATURE);
    }

    @Override
    protected void perform(Application application) {
        // Update application status
        application.setStatus(ApplicationStatus.SERVICING);

        // Get contract signature date
        LocalDate signDate = documentFinder
            .latest(application.getBorrowerParticipant(), 
                   ParticipantDocumentTypesConfiguration.APPLICATION_CONTRACT)
            .get().getDecisionMadeAt().atZone(ZoneId.systemDefault()).toLocalDate();

        // Create active credit
        ExampleCredit credit = new ExampleCredit();
        credit.setApplication(application);
        credit.setCondition(application.getCondition());
        credit.setStartDate(signDate);
        entityManager.persist(credit);

        // Initialize credit calculations
        calculationService.calculate(credit.getId(), signDate, signDate);
    }
}
----

**Complete Conversion Flow:**

1. **Offer Generation**: Participant gets approved → offers generated via `GenerateOffersParticipantAction`
2. **Offer Selection**: Participant selects offer → `SelectRegularConditionAction` creates `CreditCondition`
3. **Contract Generation**: System generates contract document for signature
4. **Contract Signature**: Participant signs contract → triggers `ContractSignChecker`  
5. **Credit Creation**: Checker creates `ExampleCredit` and initializes calculations
6. **Servicing**: Credit becomes active and ready for operations

**Key Components:**

* **ConditionForm**: Captures participant's chosen amount, term, and start date
* **CreditCondition**: Immutable terms including payment calculation and secured offer reference
* **PaymentSchedule**: Pre-calculated payment schedule based on selected terms
* **EntityChecker**: Automated credit creation triggered by contract signature
* **CreditCalculationService**: Initializes credit balances and schedules

=== Creating Procuring Engines

**1. Vehicle Collateral Engine**
```java
@Component
public class VehicleProcuringEngine implements ProcuringEngine {
    
    @Override
    public ProcuringType procuringType() {
        return MyProcuringTypes.VEHICLE;
    }
    
    @Override
    public Collection<? extends SecuredOffer> generateSecuredOffers(ProductOffer productOffer) {
        MyLoanOffer offer = (MyLoanOffer) productOffer;
        VehicleSecuredOffer securedOffer = new VehicleSecuredOffer(offer);
        
        // Apply vehicle-specific business logic
        BigDecimal baseRate = offer.getProductAdditive().getInterestRate();
        
        // Vehicle collateral typically reduces rate by 1-2%
        BigDecimal adjustedRate = baseRate.subtract(BigDecimal.valueOf(0.015));
        securedOffer.setAdjustedInterestRate(adjustedRate);
        
        // Set loan-to-value ratio (typically 80-90% for vehicles)
        securedOffer.setLoanToValueRatio(BigDecimal.valueOf(0.85));
        
        return List.of(securedOffer);
    }
}
```

**3. Property Collateral Engine**
```java
@Component
public class PropertyProcuringEngine implements ProcuringEngine {
    
    private final PropertyValuationService propertyService;
    
    public PropertyProcuringEngine(PropertyValuationService propertyService) {
        this.propertyService = propertyService;
    }
    
    @Override
    public ProcuringType procuringType() {
        return MyProcuringTypes.PROPERTY;
    }
    
    @Override
    public Collection<? extends SecuredOffer> generateSecuredOffers(ProductOffer productOffer) {
        MyLoanOffer offer = (MyLoanOffer) productOffer;
        
        // Only generate if participant has property
        if (!hasEligibleProperty(offer.getParticipant())) {
            return Collections.emptyList();
        }
        
        PropertySecuredOffer securedOffer = new PropertySecuredOffer(offer);
        
        // Property collateral gets best rates
        BigDecimal baseRate = offer.getProductAdditive().getInterestRate();
        BigDecimal adjustedRate = baseRate.multiply(BigDecimal.valueOf(0.7)); // 30% reduction
        securedOffer.setAdjustedInterestRate(adjustedRate);
        
        return List.of(securedOffer);
    }
    
    private boolean hasEligibleProperty(Participant participant) {
        // Business logic to check property ownership
        return propertyService.hasVerifiedProperty(participant);
    }
}
```

== Procuring SDK Internals

The preceding sections walked through the user-facing story of wiring a secured offer into your project. This section is the SDK reference: it enumerates every class that lives in `com.timvero.application.procuring`, the contracts they impose, and how they are discovered and wired at runtime. Read this when you need to understand _why_ a particular shape is required — not _how_ to add another procuring type.

The subsystem is intentionally small: one interface (`ProcuringEngine`), one service (`ProcuringEngineService`), one `CustomEnum` key (`ProcuringType`), one `@MappedSuperclass` entity (`SecuredOffer`), and three integration points for tabs/actions/views. That is the entire SDK surface — everything else is in your project.

=== ProcuringEngine — The Extension Point

`ProcuringEngine` is the sole extension point. Every engine declares which `ProcuringType` it handles and knows how to expand a `ProductOffer` into zero or more `SecuredOffer` instances.

[source,java]
----
public interface ProcuringEngine {

    ProcuringType procuringType();

    Collection<? extends SecuredOffer> generateSecuredOffers(ProductOffer productOffer);
}
----

Contract rules:

* `procuringType()` MUST return a stable, non-null `ProcuringType` — it is used as a `Map` key at startup, so two engines reporting the same type will cause the last one registered to silently shadow the other.
* `generateSecuredOffers(ProductOffer)` MAY return an empty collection when no secured variant applies (e.g. participant has no eligible collateral). The caller treats an empty result as "show only the unsecured offer".
* Implementations are expected to be stateless Spring beans — the `ProcuringEngineService` holds a single instance per type for the lifetime of the application context.
* The method is called on every tab render and every view assembly (see `AbstractSecuredOffersTab` and `ProductOfferViewService` below), so it MUST be cheap. Do not perform remote calls inside `generateSecuredOffers` — cache results on the offer or pre-compute during workflow.

=== ProcuringEngineService — Orchestrator

`ProcuringEngineService` is not abstract and not overridable. It is a simple dispatcher that discovers every `ProcuringEngine` bean in the context and indexes them by type:

[source,java]
----
public ProcuringEngineService(Collection<ProcuringEngine> procuringEngines) {
    this.procuringEngineMap = procuringEngines.stream()
        .collect(Collectors.toMap(ProcuringEngine::procuringType, Function.identity()));
}

public Collection<? extends SecuredOffer> generateSecuredOffers(ProductOffer offer) {
    ProcuringType procuringType = offer.getProductAdditive().getProcuringType();
    ProcuringEngine procuringEngine = procuringEngineMap.get(procuringType);
    return procuringEngine.generateSecuredOffers(offer);
}

public SecuredOffer findSecuredOffer(ProductOffer productOffer, String securedOfferKey) {
    return generateSecuredOffers(productOffer).stream()
        .filter(secured -> secured.getOfferKey().equals(securedOfferKey))
        .collect(Lang.exactlyOne());
}
----

Key behaviours:

* **Dispatch is driven by the product additive.** The engine is chosen by `offer.getProductAdditive().getProcuringType()` — not by a participant attribute, not by a runtime switch. See <<credit-product>> for how the additive stores this field.
* **`findSecuredOffer` uses `Lang.exactlyOne()`.** This is a hard assertion — if the engine returned zero matches, or more than one offer with the same key, the call throws. In practice this means every `SecuredOffer` an engine produces MUST have a unique `getOfferKey()` within the collection returned for a single `ProductOffer`.
* **There is no fallback.** If the additive references a `ProcuringType` with no registered engine, `procuringEngine.generateSecuredOffers(offer)` is a `NullPointerException`. Treat "register engine" as a first-class deployment step whenever a new `ProcuringType` is added to configuration.

=== ProcuringType & ProcuringTypeDeserializer

`ProcuringType` extends `CustomEnum<ProcuringType>` — the same pattern used for participant status, application status, and other open-ended keys. See <<entity-module-sdk-reference-hierarchy>> for the full `CustomEnum` contract; the short version is that instances register themselves into a static registry at classloading time, and equality is by name.

[source,java]
----
public class ProcuringType extends CustomEnum<ProcuringType> implements Comparable<ProcuringType> {
    public ProcuringType(String name) { super(name); }
    public static ProcuringType[] values() { return ProcuringType.values(ProcuringType.class); }
}
----

Two consequences:

* The `public static final ProcuringType PENALTY = new ProcuringType("PENALTY")` declaration is the registration — no `@PostConstruct`, no separate factory. Placing it on a `@Configuration` class guarantees the static initializer runs before anything else resolves procuring types.
* `ProcuringType.values()` returns the registry snapshot. The example project's form service uses this to populate the dropdown shown while editing a credit product additive (see `ExampleCreditProductAdditiveFormServiceImpl`, line 41).

`ProcuringTypeDeserializer` is the Jackson counterpart — plain string-to-enum lookup via the `CustomEnum` registry:

[source,java]
----
public class ProcuringTypeDeserializer extends JsonDeserializer<ProcuringType> {
    @Override
    public ProcuringType deserialize(JsonParser p, DeserializationContext ctxt) throws IOException {
        return ProcuringType.valueOf(ProcuringType.class, p.getText());
    }
}
----

Wire it with `@JsonDeserialize(using = ProcuringTypeDeserializer.class)` on any DTO field that carries a procuring type over the wire. Without it, Jackson's default enum handling will not locate your registered instances.

=== SecuredOffer Entity Structure

`SecuredOffer` is a `@MappedSuperclass` — it does not define a table on its own. Subclasses own the table, and the SDK imposes only two abstract methods:

[source,java]

* Each project declares its own `originalOffer` relationship with a concrete `ProductOffer` subtype — the SDK cannot pick that type for you.
* Each project typically wants JOINED inheritance keyed by `procuring_type` discriminator. The real base lives in the project, not the SDK.

=== AbstractSecuredOffersTab

`AbstractSecuredOffersTab` is the only opinionated UI integration in the SDK. It renders the `"/application/tab/offers"` Thymeleaf template with a pre-built map of `ProductOffer` to its secured variants:

[source,java]
----
public abstract class AbstractSecuredOffersTab<ID extends Serializable, E extends Persistable<ID>>
        extends EntityTabController<ID, E> {

    @Autowired
    private ProcuringEngineService procuringEngineService;

    @Override
    protected String getTabTemplate(ID id, Model model) throws Exception {
        E entity = loadEntity(id);
        Map<ProductOffer, List<SecuredOffer>> existingSecuredOffers = getSecuredOffers(entity).stream()
            .collect(Collectors.groupingBy(SecuredOffer::getOriginalOffer));

        Map<? extends ProductOffer, List<? extends SecuredOffer>> allOffers = getInitialOffers(entity).stream()
            .collect(Collectors.toMap(
                Function.identity(),
                offer -> existingSecuredOffers.getOrDefault(offer, Collections.emptyList())));

        model.addAttribute("allOffers", allOffers);
        return "/application/tab/offers";
    }

    protected abstract Collection<? extends ProductOffer> getInitialOffers(E entity);
}
----

Integration points for your subclass:

* **Implement `getInitialOffers(E entity)`** — return the raw `ProductOffer` list to expand. The base class will call the `ProcuringEngineService` for each.
* **Override `getSecuredOffers(E)` only if you need pre-generated offers** — by default it regenerates on every render by iterating `getInitialOffers`. Override when you persist secured offers and want to show the stored set instead.
* The template path `/application/tab/offers` is fixed — either ship a Thymeleaf template at that path or override `getTabTemplate` to point elsewhere.

=== ProcuringConfiguration

`ProcuringConfiguration` is the Spring wiring. It is declared as a `@TimveroModule` (so it is picked up by the standard Pumpkin module scan) and conditionally creates the `ProcuringEngineService` bean only when at least one `ProcuringEngine` is present:

[source,java]
----
@TimveroModule
@ComponentScan
public class ProcuringConfiguration {

    @Bean
    @ConditionalOnBean(ProcuringEngine.class)
    ProcuringEngineService procuringEngineService(Collection<ProcuringEngine> procuringEngines) {
        return new ProcuringEngineService(procuringEngines);
    }
}
----

Implications:

* **Zero engines = no service bean.** If your project has no procuring engines at all, `@Autowired ProcuringEngineService` fails at context startup. Either register at least one engine, or gate the autowire with `@Autowired(required = false)`.
* **`@ComponentScan` on this class is local** — it scans `com.timvero.application.procuring.**`. Your project engines live in your own package and are picked up by your project's component scan, not this one. The conditional bean sees all engines regardless of which scan registered them.
* **Module load order** — `@TimveroModule` guarantees the class is processed during the Pumpkin module initialization phase, so `ProcuringEngineService` is available before any tab or action that autowires it.

=== Anti-Patterns

Specific to the procuring subsystem:

* {bad} **Registering two `ProcuringEngine` beans with the same `procuringType()`** — the second silently overwrites the first in `procuringEngineMap`. There is no collision check. Split into distinct types instead.
* {bad} **Returning `SecuredOffer` instances with colliding `getOfferKey()` values** from a single engine call — `findSecuredOffer` uses `Lang.exactlyOne()` and will throw on submit. Always embed the `ProductOffer` UUID _and_ the variant tag in the key (pattern: `uuid + ":VARIANT"`).
* {bad} **Performing I/O inside `generateSecuredOffers`** — the method is called on every tab render and every view assembly pass. Pre-compute during workflow execution and cache on the offer.
* {bad} **Referencing a `ProcuringType` in a `CreditProductAdditive` with no registered engine** — produces a runtime NPE on the first offer view. Treat type registration and engine registration as a single atomic deployment step.
* {bad} **Making `SecuredOffer` an `@Entity` directly in your project base** — the SDK class is `@MappedSuperclass` specifically so your concrete base can own the `@Inheritance` strategy and discriminator column. Wrapping it in a second `@Entity` layer is not supported.

== Offer Subsystem — SDK Internals

This section is the companion reference to the user tutorial above. Everything earlier answers "how do I configure offers for my product" — this section answers "how is the offer subsystem wired internally, and what exactly can I extend". Read this when you are subclassing `ProductOffer`, registering a new `OfferEngineDataProcessor`, writing scripts that bind unusual objects, or diagnosing why a descriptor is not being picked up.

Scope: the `com.timvero.loan.offer` package (23 files, 8 subpackages) plus the external `com.timvero.loan.offer_engine_input` marker (1 file). The procuring subsystem — a related but separate extension axis — is documented in <<procuring-sdk-internals>>; do not confuse `OfferEngine` (the generator of `ProductOffer` instances) with `ProcuringEngine` (the per-offer variant expander). An `OfferEngine` runs once per product/additive/profile combination; a `ProcuringEngine` runs on each already-generated `ProductOffer` to materialize collateral-bound variants.

The offer subsystem is the bridge between <<credit-product>> (product/additive templates) and <<credit-servicing>> (the `CreditCondition` consumed at servicing time). Scripts executed here can read facts produced during <<flowable-internals>> workflows and consume metrics from the <<feature-service>>.

=== ProductOffer — The Entity

`ProductOffer` is the concrete output of the generation pipeline. It is a `@MappedSuperclass`, not an `@Entity`:

[source,java]
----
@MappedSuperclass
public abstract class ProductOffer extends BaseEntity<Long> {

    @ManyToOne(fetch = FetchType.EAGER, optional = false)
    @JoinColumn(name = "product_additive", nullable = false)
    @Audited(targetAuditMode = RelationTargetAuditMode.NOT_AUDITED)
    private CreditProductAdditive productAdditive;

    protected ProductOffer() { }

    @Transient
    public CreditProduct getCreditProduct() {
        return getProductAdditive().getProduct();
    }

    @Override
    public abstract String toString();
}
----

Non-obvious facts:

* **It is abstract _and_ a `@MappedSuperclass`.** Your project must declare a concrete `@Entity` subclass — typically one per credit product family. The SDK never persists `ProductOffer` directly; see <<entity-module-sdk-reference-hierarchy>> for the base hierarchy.
* **`toString()` is abstract.** This is a deliberate contract: the bean wrapper in `OfferEngine.fillOffer` and several UI views call `toString()` for log lines and ActionHistory entries. An offer with the default `Object.toString()` produces useless identity hashes in the audit log.
* **`productAdditive` is eagerly fetched and non-nullable.** The engine sets it in `produceOffers` before the script runs — scripts observe an offer that already has its additive wired, which is why `offer.productAdditive.product` is legal inside a script.
* **`@Audited(targetAuditMode = NOT_AUDITED)`** — Envers records the FK value but does not snapshot the full `CreditProductAdditive` tree. If you audit the additive separately it is double-counted; if you do not, you still see which additive was chosen at offer time.
* **ID type is `Long`, not `UUID`.** Offers are cheap row-per-generation artifacts; UUIDs are reserved for long-lived or externally-referenced entities. Do not try to "upgrade" the ID type — several downstream queries (procuring, credit conversion) are keyed by `Long` offer IDs.

The pattern for a project subclass:

[source,java]
----
@Entity
@Table(name = "mortgage_offer")
@DiscriminatorValue("MORTGAGE")
@AuditOverride(forClass = ProductOffer.class)
public class MortgageOffer extends ProductOffer {

    @Column(name = "amount")
    private MonetaryAmount amount;
    @Column(name = "interest_rate", precision = 10, scale = 6)
    private BigDecimal interestRate;
    // ... other fields driven by the script

    @Override
    public String toString() {
        return "%s @ %s for %d months".formatted(amount, interestRate, termMonths);
    }
}
----

The concrete class also needs a `createOffer()` factory on the matching `CreditProductAdditive` subclass — that is the method `OfferEngine` invokes, not a constructor.

=== OfferEngine — The Extension Point

`OfferEngine` is the orchestrator — a `@Service` with two public entry points. The contract:

[source,java]
----
@Transactional(propagation = Propagation.MANDATORY)
public <ID, E extends Persistable<ID>> Collection<? extends ProductOffer> generateOffers(
    OfferEngineDataProcessor<ID, E> processor,
    Collection<? extends CreditProduct> products,
    E entity) throws OfferGenerationException;

@Transactional(rollbackFor = OfferGenerationException.class)
public <ID, E extends Persistable<ID>> List<Object> testOffers(
    Expression expression, UUID productId, Long additiveId,
    ExecutionResultType resultType, String entityId) throws OfferGenerationException;
----

`generateOffers` is what the origination workflow calls; `testOffers` is what the descriptor "Evaluate" button calls.

Non-obvious facts:

* **`Propagation.MANDATORY` on `generateOffers`.** It **will throw** if you call it without an outer transaction. The workflow service task that invokes it is responsible for opening the transaction — if you are calling it from a Quartz job or a custom controller, wrap it in `@Transactional` yourself.
* **Product → engine resolution is by `ExecutionResultType`, not by product.** `product.getOfferEngineTypes()` is a set; the engine iterates additives and picks the descriptor whose `executionResultType` matches. A single product can participate in multiple offer pipelines keyed by different result types (e.g. `ORIGINATION` vs `RENEWAL`).
* **`findEngineDescriptor` queries by `lineageId + executionResultType + active = true`.** The additive stores `offerEngines: Map<ExecutionResultType, UUID>` where the UUID is the _lineage_ id (stable across versions), not the current descriptor's primary key. Saving a new descriptor version keeps the additive's pointer valid — that is why descriptor editing is non-breaking.
* **Script result shapes.** `produceOffers` accepts three shapes from the script:
  ** `return offer;` (the bound `offer` object, mutated) — returned as-is.
  ** `return { amount: 1000, term: 12 };` — keys set via `BeanWrapper` reflection on the offer.
  ** `return null;` or anything else — treated as "no offer for this profile", silently skipped.
* **`MonetaryAmount` coercion is a special case.** `fillOffer` hard-codes this: if the target property is `MonetaryAmount`, the source is a `Number`, and the offer itself implements `CurrencySupplier`, it wraps the number in the offer's currency. Scripts that set amounts as plain numbers only work when the offer exposes a currency.
* **`OfferGenerationException` wraps the cause twice.** It stores the original exception both as the `Throwable` cause and as a separately-accessible `getException()` field, so the controller's exception handler can show the root message without walking `getCause()`. Do not throw it directly from a script — the engine will catch any `ScriptException` or generic `Exception` and wrap it with the descriptor id/name.

=== Offer Script Execution

Scripts run under `ScriptManager` (a separate SDK) but the bindings are controlled entirely by `OfferEngine.getBindings`:

[source,java]
----
private Map<String, Object> getBindings(Map<String, Object> profile, CreditProductAdditive additive,
    ProductOffer offer, Object entity, boolean persist) {
    Map<String, Object> bindings = new LinkedHashMap<>();
    bindings.put("product", additive.getProduct());
    bindings.put("productAdditive", additive);
    bindings.put("offer", offer);
    bindings.put("profile", profile);
    bindings.put("entity", entity);
    if (entity instanceof HasMetric subject) {
        bindings.put("metrics", new MetricContainer(subject, metricService, persist));
    }
    return Collections.unmodifiableMap(bindings);
}
----

The five always-present bindings and the conditional `metrics` binding are the full surface area a script sees. Non-obvious details:

* **`metrics` binding only appears when the entity implements `HasMetric`.** A script that references `metrics.get(...)` for a non-`HasMetric` entity fails with a binding lookup error, not a null pointer. This is the gate between an origination entity and the <<feature-service>>.
* **`MetricContainer.get(mappingName)` is cached per call.** Multiple lookups of the same metric inside one script invocation hit the in-memory `ConcurrentHashMap`, not the DB. But the cache is per-container, so each script run on each profile pays the first-lookup cost once.
* **`persist` is `false` in `testOffers`, `true` in `generateOffers`.** The "Evaluate" button in the descriptor UI computes metrics transiently, so you can experiment without creating audit-visible metric rows. The production pipeline persists.
* **Exception metrics throw through the script.** If a metric's `dataType` is `FeatureDataType.EXCEPTION`, `MetricContainer.get` throws a `RuntimeException` with the metric's content. The engine catches it, wraps it in `OfferGenerationException`, and aborts the additive — not the whole product. Other additives still generate.
* **`product` and `productAdditive` are live JPA entities.** Scripts can navigate lazy associations _inside_ the script's transaction (generation is `Propagation.MANDATORY`), but storing a reference in a long-lived object will cause `LazyInitializationException` later.
* **The bindings map is `unmodifiableMap`.** Scripts cannot add their own bindings; all mutation must go through the `offer` object.

The script engine itself is chosen at descriptor creation time (JS via GraalVM, SpEL, or any other registered `ScriptEngineConfigurator`). See `InterestRateServiceConfigurator` below for how a function is exposed to both engines simultaneously.

[[offer-engine-data-processor]]
=== OfferEngineDataProcessor — Entity-to-Profile Bridge

`OfferEngineDataProcessor` is the SPI that teaches `OfferEngine` how to turn an origination entity into a _collection_ of script profiles:

[source,java]
----
public abstract class OfferEngineDataProcessor<ID, A extends Persistable<ID>>
    extends EntityProcessor<ID, A> {

    public abstract ExecutionResultType getResultType();
    public abstract Collection<Map<String, Object>> mapToData(A entity);
    protected void fillOffer(ProductOffer offer, Map<String, Object> data) {}
}
----

Three concrete obligations:

* **`getResultType()`** — the `ExecutionResultType` this processor handles. The engine builds a `Map<ExecutionResultType, OfferEngineDataProcessor>` at startup and **throws on duplicate types**. Two processors for the same type is a hard error.
* **`mapToData(entity)`** — returns a `Collection<Map<String, Object>>`. Each map becomes one `profile` binding, and produces _at most one_ offer (the script can return empty to drop it). A single `generateOffers` call can therefore emit N offers per additive where N = `profiles × valid script results`. Use this to fan out offers — e.g. one per term duration, one per borrower, one per scenario.
* **`fillOffer(offer, data)` (optional hook)** — called _after_ the script evaluates and before the offer is added to the result list. Use this to copy profile-identifying fields onto the offer (e.g. which scenario produced it) that you do not want the script author to be responsible for setting.

The `EntityProcessor` superclass also provides `getEntityClass()` which the engine uses to look up the entity via `ValidationEntitySearchService.loadEntity` during "test offer" evaluation.

Binding order: `mapToData` → `createOffer` → `script.eval(bindings)` → `fillOffer` (script-mutation) → `processor.fillOffer` (post-processing) → collect. If your script sets a field and the processor's `fillOffer` also sets it, the processor wins.

=== OfferEngineDescriptor — The Persisted Script

`OfferEngineDescriptor` is the DB row users edit in the `/offer-engine-descriptor` UI. It extends `HistoryEntity`, which means descriptors are versioned by `lineageId`:

[source,java]
----
@Entity
@Table(name = "offer_engine_descriptor")
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
@HideInDocs
public class OfferEngineDescriptor extends HistoryEntity implements NamedEntity {
    private String name;
    @Embedded private Expression expression;
    @Type(value = NamedParameterType.class)
    private ExecutionResultType executionResultType;
}
----

Non-obvious facts:

* **`@HideInDocs`** — the generated data-model documentation _skips_ this entity. It is a runtime concern, not a model concern.
* **Second-level cache is `READ_WRITE`.** The descriptor is hit on every offer generation; cache it or the engine hammers the DB. `DisableOfferEngineDescriptorAction` explicitly calls `emf.getCache().evict(...)` after deletion because of this.
* **Versioning via `HistoryEntity`.** `findByLineageIdAndExecutionResultTypeAndActiveTrue` is the primary lookup — the current "active" version of a descriptor lineage. Editing through `OfferEngineDescriptorFormService` creates a new row and flips the active flag; old rows remain for audit.
* **`executionResultType` is immutable after creation.** `OfferEngineDescriptorFormService.save` enforces this with a `ValidationException` on update attempts. If you need to change the type, create a new descriptor — the additive's pointer will need to be re-aimed.
* **The `Expression` is embedded, not a relation.** `Expression` from `com.timvero.loan.risk.scripting.expression.entity` holds the script body + engine name inline; there is no separate expression table.

Repository extras:

* `existsByLineageIdAndExecutionResultTypeAndActiveTrue` — pre-flight check used by the product additive form to validate the user's selection.
* `findByExecutionResultType(Collection<ExecutionResultType>)` — bulk lookup via criteria API, filtered to `active = true`. Used when rendering drop-downs.
* `getNameByLineageId` — a native query that fetches the most recent name without loading the full row. Used in audit trails and log lines where only the human-readable name matters.

=== OfferEngineInput — The External Marker

The `com.timvero.loan.offer_engine_input` module contains exactly one file:

[source,java]
----
package com.timvero.loan.offer_engine_input;

public class OfferEngineInput {
}
----

This is intentionally empty. It is a _module boundary marker_: the package exists as a separate Maven module so that code wishing to produce inputs for an offer engine can depend on `offer_engine_input` without pulling in the full `origination` module. In practice:

* {ok} Domain modules that expose facts consumable by offer scripts (metrics, external data providers, workflow outputs) declare a dependency on `offer_engine_input` and place their input-DTO classes in this package.
* {ok} The `origination` module, which contains `OfferEngine` itself, has `offer_engine_input` as a compile-time dependency — the runtime bindings map (`entity`, `profile`, etc.) can reference types declared downstream.
* {bad} Putting your project's entity classes here "to make them available to the engine" — the engine does not scan this package. The binding mechanism is reflection on the script's namespace, not classpath scanning. You do not need to put anything here unless you are splitting your monorepo into offer-input and offer-consumer modules.

Treat `OfferEngineInput.java` as a sentinel: its presence confirms the module is loaded, and its package is where shared input-DTO contracts belong. It is not intended to be extended or instantiated.

=== Action Layer — action/ subpackage

Four controllers wire the descriptor entity into the standard Pumpkin entity-action framework:

* **`CreateOfferEngineDescriptorAction`** — `EntityCreateController`. `isOwnPage()` returns `true`, so create lives on its own route rather than inside a modal.
* **`EditOfferEngineDescriptorAction`** — `EditEntityActionController`. `BTN_SUCCESS` highlight.
* **`ActivationOfferEngineDescriptorAction`** at `/activation`, order `16700`. Guard: `when(template -> !template.isActive())`. Activation is delegated to `formService.activate(id)` — the `HistoryEntityFormService` base handles the "deactivate old version, activate new" atomic flip.
* **`DisableOfferEngineDescriptorAction`** at `/disable`, order `29001`. Guard: `when(OfferEngineDescriptor::isActive)`. It deletes the row via the repository and **manually evicts** `OfferEngineDescriptor` from the second-level cache via `emf.getCache().evict(OfferEngineDescriptor.class, id)`. This manual eviction is the safeguard; JPA's cascade-evict is not guaranteed to fire on `delete`. Shows a `yes-no` confirmation template.

Non-obvious: there is no separate "create draft and submit" flow — activation is a first-class action separate from edit. A descriptor is created inactive, edited freely, then activated atomically.

=== Form Layer — form/ subpackage

Three pieces:

* **`OfferEngineDescriptorForm`** — plain DTO. `@NotBlank name`, `@Valid @NotNull expression`, `@NotNull executionResultType`.
* **`OfferEngineDescriptorFormMapper`** — MapStruct mapper between form ↔ entity. Explicitly ignores `lineageId` on update so the version chain is never accidentally broken through the form.
* **`OfferEngineDescriptorFormService`** extends `HistoryEntityFormService`. Three notable behaviors:
  ** `assembleEditModel` defaults a new form's engine to `JSScriptEngine.NAME` so the UI opens on JS (most common choice).
  ** It populates `executionResultTypes`, `creditProducts` (filtered to state = ACTIVE), `engineNames`, and `entityTypes` for the form. `entityTypes` maps result type → entity class so the UI can show which entity each script operates on.
  ** `save(id, form)` **enforces `executionResultType` immutability** as described above — changing it on an existing descriptor rejects the form with `offerEngineDescriptor.unmodifiableField`.
  ** `additiveByProduct(productId)` returns the list of additive IDs for a given product — used by the cascading dropdown in the "Evaluate" panel.
  ** `getOfferEngineDescriptorVersions(descriptor)` returns the full version chain sorted by `createdAt` descending — backs the history tab.

=== Controller — controller/ subpackage

`OfferEngineDescriptorController` at `/offer-engine-descriptor`, registered in the settings menu (`@MenuItem(order = 900_120, name = "offerEngineDescription", group = "setting")`):

* **`POST /evaluate-expression`** — the "try it" endpoint. Takes an `OfferEngineEvaluationForm` (product, additive, result type, entity id, expression body), runs `offerEngine.testOffers`, and returns a `List<String>` human-readable renderings. Offers are serialized via `BeanWrapper` introspection with `productAdditive` _explicitly_ filtered out (otherwise the Jackson roundtrip would blow up on the bidirectional association).
* **`GET /additiveByProduct/{productId}`** — feeds the cascading dropdown.
* **`GET /entities?search=...&resultType=...`** — search-as-you-type endpoint for picking a test entity. Gated by `@PreAuthorize("hasRole(this.viewRole)")`.
* **`@ExceptionHandler(OfferGenerationException.class)`** — returns `412 Precondition Failed` with the localized message. The evaluation UI expects this specific status and renders the message inline instead of as a global toast.

=== Filter, Tab — filter/ + tab/ subpackages

Minimal infrastructure:

* **`OfferEngineDescriptorFilter`** extends `ListFilter`. Only one filter field: `@Field(restriction = EQ, value = "active")`, defaulting to `true`. So the list view shows active descriptors by default; toggle the filter to see the archive.
* **`OfferEngineDescriptorFilterService`** — empty subclass of `FilterListService`, registered as `@Service` so DI picks it up.
* **`DescriptorDetailsTab`** at `/details`, order `1000`. Always visible. It populates `engineNames`, `descriptor`, `history`, and `isNoneActiveVersionExist` (true when no version of this lineage is currently active — drives the "activate" button visibility).

=== Export — export/ subpackage

Five classes implement the import/export of descriptors (used for environment promotion and project templates):

* **`OfferEngineDescriptorExporter`** — `@Service`, `@Order(7300)`, extends `AbstractHistoryEntityExporter`. Version string is `"1"`; bumping this invalidates older export files.
* **`OfferEngineDescriptorToDtoMapper`** — MapStruct mapper to/from `OfferEngineDescriptorExportData`. Ignores `lineageId` on import (a fresh lineage is minted in the target environment).
* **`OfferEngineDescriptorExportData`** — the export shape. Includes `name`, `expression` (as `ExpressionForm`, not the embedded entity — fewer JPA concerns on deserialize), and `executionResultType`.
* **`ExecutionResultTypeDeserializer`** / **`ExecutionResultTypeKeyDeserializer`** — Jackson hooks that reconstruct `ExecutionResultType` from a plain string. `ExecutionResultType` is a value-class wrapper around a name; without these, Jackson would try to treat it as a bean. The _key_ variant exists because additives expose `Map<ExecutionResultType, UUID> offerEngines`, which JSON encodes as an object whose keys must be deserialized separately from the values.

=== InterestRateServiceConfigurator — Script Function Exposure

`InterestRateServiceConfigurator` is a `@Component` that implements both `PolyglotScriptEngineConfigurator` (for GraalVM/JS) and `SpelScriptEngineConfigurator` (for SpEL) to expose a single function, `interestRate(expression, date)`, to all scripts:

[source,java]
----
@Override
public void configContext(StandardEvaluationContext context) {
    context.registerFunction("interestRate", interestRateMethod);
}

@Override
public void configContext(SimpleBindings globalBindings) {
    globalBindings.put("interestRate", interestRateFunction);
}
----

Non-obvious:

* **The date type differs per engine.** SpEL gets a plain `LocalDate`; GraalVM JS passes a polyglot `Value` (often a JS `Date`), so the configurator wraps the function in `toLocalDate(date)` conversion. Write scripts assuming `interestRate(rateName, date)` works without thinking about dates — the configurator handles both calls transparently.
* **Errors are wrapped in a `RuntimeException`.** A `RateEvaluationException` from `LoanRateService` becomes a generic runtime error in the script; the script engine catches it, and the `OfferEngine` wraps it in `OfferGenerationException`. If a rate name is missing, the engine halts that additive only.
* **This is the integration point with the rate curve system.** If you need to expose more functions globally to every offer script (e.g. a tax calculator, a fee lookup), add a `@Component` that implements the two configurator interfaces and registers the function. There is no other approved extension point.

=== Anti-Patterns

Specific to the offer subsystem (additional to the procuring list above):

* {bad} **Registering two `OfferEngineDataProcessor` beans with the same `ExecutionResultType`** — `OfferEngine.setDataServices` throws on duplicate types. Split the type instead of overloading one.
* {bad} **Calling `OfferEngine.generateOffers` outside a transaction** — the `MANDATORY` propagation means no outer TX = immediate exception. Wrap the caller in `@Transactional` or invoke it from a workflow service task (which already runs transactionally).
* {bad} **Putting I/O or remote calls inside `OfferEngineDataProcessor.mapToData`** — it runs on every offer generation pass, once per product × additive. Pre-compute into the origination entity and only _read_ here.
* {bad} **Scripts that return a `Map` with keys not present on the offer** — `BeanWrapper.getPropertyDescriptor(key)` returns `null`, causing an NPE. The engine does not silently ignore unknown fields; validate keys match offer properties.
* {bad} **Storing an `OfferEngineDescriptor` reference in a field of another entity** — the descriptor's second-level cache and version flipping make aged references unreliable. Store the `lineageId` + `executionResultType` pair and resolve via the repository on demand.
* {bad} **Extending `ProductOffer` without overriding `toString()`** — compiles fine because the abstract method is inherited; fails at audit-log time when the default `Object.toString()` is called.
* {bad} **Changing `executionResultType` on an existing descriptor by editing the DB row directly** — the form service guards this in code; a DB-level edit bypasses the guard but leaves additive pointers dangling. Always create a new descriptor instead.
