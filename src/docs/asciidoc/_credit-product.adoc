[[credit-product]]
= Credit Product Module — entity model, lifecycle & rates

:sourcedir: ../../main/java/com/timvero/example/admin
:libdir: ../../main/java/com/timvero/loan/product

The Credit Product module defines loan product templates — amount ranges, terms, interest rates, and fee structures — that feed into the Offer Engine to generate personalized offers. This guide covers the entity model, state lifecycle, rate evaluation service, form validation, and filtering. For web layer setup (controllers, actions, tabs, templates), see <<product-web-components-configuration>>.

== What You Need to Know

A credit product is a **loan template** — not a loan itself. It defines the boundaries (min/max amounts, terms, fees) within which individual offers are generated for participants. The hierarchy:

----
CreditProduct (template)
  └── CreditProductAdditive (specific configuration)
        └── ProductOffer (personalized for participant)
              └── SecuredOffer (with collateral/procuring)
                    └── CreditCondition (selected terms)
                          └── Credit (active loan)
----

Key characteristics:

- Products use **single-table inheritance** — your subclass adds columns to `credit_product`
- Products have a **three-state lifecycle**: DRAFT -> ACTIVE -> INACTIVE
- Each product has one or more **additives** that define interest rates and offer engine mappings
- The **LoanRateService** evaluates SpEL expressions with reference rate bindings for variable-rate products
- Products are **cached** (Hibernate L2 cache, READ_WRITE strategy)

== Rate Evaluation

=== LoanRateService

The `LoanRateService` evaluates interest rate expressions using Spring Expression Language (SpEL). It supports both fixed rates and variable rates that reference external rate indices.

.How it works
----
Expression: "SOFR + 0.025"
                │
    ┌───────────┘
    ▼
ReferenceRateService.getOnDate(date)
    → { "SOFR": 0.053, "PRIME": 0.085, ... }
    │
    ▼
Replace abbreviations with bindings
    → "rate12345 + 0.025"
    │
    ▼
SpEL evaluation
    → 0.078 (BigDecimal)
----

=== Rate Expression Syntax

Rate expressions are standard SpEL with reference rate abbreviations injected as variables:

.Expression examples
[cols="2,3", options="header"]
|===
|Expression |Meaning

|`0.089`
|Fixed 8.9% rate

|`SOFR + 0.025`
|SOFR + 2.5% spread

|`PRIME * 0.5 + 0.01`
|Half of prime rate + 1%

|`T(java.lang.Math).max(SOFR + 0.03, 0.05)`
|Floor of 5% on SOFR + 3%
|===

Reference rate abbreviations (like `SOFR`, `PRIME`) are replaced at evaluation time with their current values from the `ReferenceRateService`.

=== Point-in-Time vs Range Evaluation

The service provides two evaluation modes:

**Single date** — `getLoanRateValueByDate(expression, date)`:

Returns the evaluated rate for a specific date. Used when calculating interest for a single day or when displaying the current rate.

**Date range** — `getLoanRateValueByRange(expression, from, to)`:

Returns a `NavigableMap<LocalDate, BigDecimal>` containing only the dates where the rate changed. Used for variable-rate credits where the interest rate shifts when reference rates change.

[source,java]
----
// Fixed rate — returns single entry
NavigableMap<LocalDate, BigDecimal> fixed = rateService
    .getLoanRateValueByRange("0.089", startDate, endDate);
// → { startDate: 0.089 }

// Variable rate — returns entries at each change point
NavigableMap<LocalDate, BigDecimal> variable = rateService
    .getLoanRateValueByRange("SOFR + 0.025", startDate, endDate);
// → { 2026-01-01: 0.078, 2026-02-15: 0.081, 2026-03-20: 0.076 }
----

[NOTE]
====
Parsed SpEL expressions are cached in a `ConcurrentHashMap` for performance. The cache is unbounded — in practice this is fine because the number of distinct rate expressions is small (one per product).
====

== Integration Points

The product module connects to the rest of the platform through these flows:

.Product → Credit pipeline
----
CreditProduct + Additive
    │
    ▼ (OfferEngine evaluates scripts)
ProductOffer (personalized)
    │
    ▼ (ProcuringEngine adds collateral)
SecuredOffer
    │
    ▼ (Participant selects offer)
CreditCondition (locked terms)
    │
    ▼ (Contract signed → EntityChecker)
Credit (active loan servicing)
----

**Key integration classes:**

- `ScheduledEngine` — product's `engineName` references a Spring bean that calculates payment schedules
- `OfferEngineDataProcessor` — maps participant data for offer script evaluation
- `ProcuringEngine` — generates secured offer variants based on additive's `procuringType`
- `CreditCondition` — captures the selected offer's terms, including product references
- `LoanRateService` — evaluates rate expressions for variable-rate products during servicing

=== Root — `CreditProductService` and `ProductWebConfigurationSupport`

==== `CreditProductService` — transactional state flips

A minimal, deliberately-thin service that flips `CreditProductState` by calling `entityManager.getReference()`. It does **not** load the full entity — it only acquires a proxy and sets the state. This avoids loading the `additives` collection just to change a single column.

[source,java]
----
@Service
public class CreditProductService {
    @Autowired
    private EntityManager entityManager;

    @Transactional
    public void deactivate(UUID id) {
        entityManager.getReference(CreditProduct.class, id).setState(INACTIVE);
    }

    @Transactional
    public void activate(UUID id) {
        entityManager.getReference(CreditProduct.class, id).setState(ACTIVE);
    }
}
----

NOTE: Because transitions bypass Hibernate dirty-checking in the usual edit flow, `EnableCreditProductAction` calls `reloadPageHelper.reload(...)` via `AddCreditProductAdditiveAction` — the browser must refresh to see the new state. Callers of `CreditProductService` directly must do the same.

==== `ProductWebConfigurationSupport` — programmatic bean registry

Instead of a dozen `@Bean` declarations, `ProductWebConfigurationSupport` implements `BeanDefinitionRegistryPostProcessor` and registers every controller/action/tab with explicit generics via `ResolvableType.forClassWithGenerics(...)`. This is load-bearing: Spring's dependency resolution uses the resolvable type to match `CreditProductAdditiveFormService<ExampleCreditProductAdditive, ExampleCreditProductAdditiveForm>` against the injection point in each action.

=== Entity package — `product.entity`

==== `CreditProduct` — single-table inheritance + L2 cache

[source,java]
----
@Entity
@Table(name = "credit_product")
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE, includeLazy = false)
public abstract class CreditProduct extends AbstractAuditable implements NamedEntity { ... }
----

Load-bearing details:

- `InheritanceType.SINGLE_TABLE` — your subclass columns all live in `credit_product`. No joins, no discriminator column override (class name is the default).
- `@Cache(... includeLazy = false)` — L2 cache does not include lazy-loaded collections; `additives` is loaded fresh per session.
- `@SQLRestriction("active = true")` on the `additives` collection is applied by Hibernate at SQL level, **not** by the JPA provider. If you write HQL joining additives, the restriction still applies. Native SQL bypasses it.
- `creditType` uses `@Type(CustomEnumType.class)` — values are persisted as VARCHAR and looked up from the `CustomEnum` registry. Do not try to use `@Enumerated(EnumType.STRING)` here; `CreditType` is not a Java enum.
- `offerEngineTypes` uses `@Type(NamedParameterType.class)` stored in `credit_product_offer_engine_type` — an element-collection, one row per supported engine type.

==== `CreditProductAdditive` — soft-delete via `UndeletableEntity`

[source,java]
----
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
public abstract class CreditProductAdditive extends UndeletableEntity<Long> implements NamedEntity {

    @ManyToOne(fetch = FetchType.EAGER, optional = false)
    @JoinColumn(name = "product_id", updatable = false)
    @Fetch(FetchMode.SELECT)
    private CreditProduct product;

    @ElementCollection(fetch = FetchType.LAZY)
    @CollectionTable(name = "credit_product_additive_offer_engine_map", ...)
    @MapKeyType(NamedParameterType.class)
    private Map<ExecutionResultType, UUID> offerEngines;

    public abstract ProductOffer createOffer();
}
----

Inheriting `UndeletableEntity<Long>` gives you the `active` column and `setActive(boolean)` — the actions toggle this flag instead of running `DELETE`. The base exposes `setActive` as `protected`; `CreditProductAdditive` re-declares it `public` so activate/deactivate actions can call it without reflection.

The `product` association is `EAGER + FetchMode.SELECT` — this avoids N+1 at the cost of an extra query per additive load. For bulk additive processing (e.g., scheduled jobs), fetch with an explicit `JOIN FETCH`.

`createOffer()` is the **only** abstract method and the single extension point for offer generation. The offer engine layer calls `additive.createOffer()` to get a pre-wired `ProductOffer` subclass — your implementation must set back-references (`offer.setAdditive(this)`, currency, etc.). See <<offer-engine>> for how the returned offer flows through engine execution.

`CreditProductAdditiveRepository` extends both `JpaRepository` and `SynchronousAccessRepository` (the latter gives pessimistic-lock helpers used by servicing). It also adds a native update to soft-delete additives without loading them:

Two types that look similar but serve different roles:

=== Filter package — `product.filter`

`BaseCreditProductFilter` is a one-field abstract ancestor. It uses the `@Field(restriction = Restriction.IN, value = "state")` annotation on a `CreditProductState[]` array — the framework expands this into a JPA `IN` clause when the filter is applied. The filter is abstract because `ListFilter` requires concrete classes for Spring's controller binding.

The SDK intentionally stops at `state`. All other filterable fields (title LIKE, credit type EQ, amount ranges) are project-specific and belong in your subclass.

=== Rate package — `product.rate`

==== `LoanRateService` — how SpEL and reference rates interact

The service pre-processes the raw expression (e.g., `"SOFR + 0.025"`) before handing it to SpEL:

. Fetch all `ReferenceRate` rows valid on `date` via `ReferenceRateService.getOnDate(date)`.
. For each reference rate, compute a synthetic key: `"rate" + (abbreviation.hashCode() & 0xFFFFFFFFL)`.
. Replace every occurrence of the abbreviation in the raw text with that key (e.g., `"SOFR + 0.025"` becomes `"rate123456 + 0.025"`).
. Store `{key -> BigDecimal}` in a `LinkedHashMap` and wrap it in a `StandardEvaluationContext` with a `MapAccessor`.
. Cache the parsed `Expression` under the transformed text in a `ConcurrentHashMap<String, Expression>`.
. Evaluate; the result must be `BigDecimal` or a `Number` coercible via `new BigDecimal(result.toString())`.

WARNING: Because the key depends on `hashCode()`, two reference rates whose abbreviations collide (astronomically unlikely but possible) would bind to the same variable. If you see `RateEvaluationException("Rate 'X' is not defined")` intermittently, check for abbreviation collisions or stale cached expressions after renaming a rate.

==== Range evaluation picks only change points

`getLoanRateValueByRange(expression, from, to)` calls `referenceRateService.getOnRangeChanges(from, to)` — a sparse map keyed by the date of every rate change. The service re-evaluates the expression on each change date, compares the new result to the previous value, and only records the entry when the result actually changes. This keeps the returned `NavigableMap` minimal — it contains change points, not every day in the range.

This is what the servicing layer uses to drive variable-rate interest accrual. See <<credit-servicing>> and <<servicing-engines>> for how the returned map feeds `ScheduledEngine.rebuildScheduleFromDate(...)`.

==== `RateEvaluationException` — user-visible vs. internal message

The exception carries **two** messages: `message` (logs/stack traces) and `visibleMessage` (UI). When `printLoanRateValue(expressionTxt)` catches the exception, it returns `e.getMessage()` — which should not be rendered raw to end users. The Thymeleaf layer should call `getVisibleMessage()` when surfacing rate-test failures in the product edit form.

==== `CopyProductAction` — two-stage copy

Copy is the only way to create a new product revision after activation. The action:

. Uses `formService.getMapper().toForm(entity)` to snapshot the current product into a form DTO.
. Overwrites only `code` and `title` from the `CopyProductForm` submitted by the user.
. Saves the new product (state defaults to DRAFT).
. Iterates active additives, converts each to an additive form via `additiveFormService.getMapper().toForm((A) additive)`, then saves with `(f, a) -> a.setProduct(product)` so the back-reference points at the new parent.

Inactive additives are intentionally skipped. If you want to propagate them, override `copyEntity(...)`.

NOTE: `CopyProductAction` does **not** currently set `copiedFrom` on the new product. If you need audit lineage, override `copyEntity` and set `entityForm.setCopiedFrom(entity.getId())` before saving.

==== `AddCreditProductAdditiveAction` — bypasses auto-reload

The action body is:

[source,java]
----
formService.save(null, f, (f1, a) -> a.setProduct(p));
reloadPageHelper.reload(p); // no auto reload because of direct DB change
----

The inline comment matters — the standard entity-action flow reloads the page automatically when the edited entity mutates. Here the mutation is on a **child** (the new additive), so the framework has no signal that the parent view needs to refresh. `ReloadPageHelper.reload(...)` manually triggers it.

If you write your own parent-child actions and notice a stale page after save, this is the hook you're missing.

==== Additive `ActivateAdditiveAction` has no confirmation dialog

Compare `DeactivateCreditProductAction` (renders `/common/action/yes-no` with a confirmation prompt) with `ActivateAdditiveAction` (no `getActionTemplate` override — one-click activation). Low-risk state flips skip the dialog; risky ones (disable product, hard-delete, etc.) require it. Follow this convention in custom actions.

=== Tab package — `product.tab`

Both tabs extend `EntityTabController<UUID, E>` and register with `@Order` — 1000 for details, 2000 for additives, so Details renders first. Both default `isVisible(entity)` to `true` — override to hide tabs for certain product subtypes (e.g., show "Additives" only when `creditType != TEMPLATE`).

`CreditProductAdditivesTab` wraps the additive collection in `Lazy.of(...)` so the database round-trip is deferred until the template actually iterates. This matters when the tab is one of several on a busy product detail page — switching tabs should not preload all tab contents.

== Best Practices

[.unstyled]
* {ok} **Define CreditType constants in a `@Configuration` class** — ensures registration at startup
* {ok} **Keep additive ranges within product ranges** — use `AdditiveFormValidator` pattern for enforcement
* {ok} **Use `@SQLRestriction` for soft-delete filtering** — the SDK already does this for additives; follow the same pattern for custom soft-delete entities
* {ok} **Mark currency as `updatable = false`** — changing currency after product creation would corrupt existing offers
* {ok} **Use `CopyProductAction` for product revisions** — copy to DRAFT, modify, activate the new version
* {ok} **Populate form model data in `assembleEditModel()`** — keeps controller logic clean and reusable
* {ok} **Register all components via `ProductWebConfigurationSupport`** — one configuration class replaces 12+ manual bean definitions
* {ok} **Prefer `EntityManager.getReference()` for state flips** — `CreditProductService` avoids loading the full product and its `additives` collection just to change one column
* {ok} **Override `getXxxTemplatePath()` instead of subclassing actions** — changing a template location is a one-line override, not a new `@Controller`
* {ok} **Use `@Order` values between SDK landmarks (16700, 19000, 20000, 25000)** — inserts custom actions predictably into the detail-page button row
* {ok} **Wrap tab collections in `Lazy.of(...)`** — follows `CreditProductAdditivesTab`'s pattern; defers DB work until the template iterates
* {ok} **Surface `RateEvaluationException.getVisibleMessage()` in the UI** — never render `getMessage()` (contains expression text and internal state)

== Anti-Patterns

[.unstyled]
* {bad} **Editing ACTIVE product fields directly via SQL** — bypasses state protection, corrupts offers and credits referencing the product
* {bad} **Forgetting `@Configuration` on CreditType definitions** — types won't register, `CreditType.values()` returns empty
* {bad} **Storing calculated rates in the product entity** — rates change over time; store the expression and evaluate at runtime via `LoanRateService`
* {bad} **Creating additives without offer engine mappings** — offer generation will silently skip the additive, producing no offers
* {bad} **Using `EAGER` fetch on the `additives` collection** — the SDK uses `LAZY` with `@SQLRestriction` intentionally; eager fetch loads all additives (including inactive) without the filter
* {bad} **Validating additive bounds only in the frontend** — always enforce server-side via `Validator` implementation; the frontend can be bypassed
* {bad} **Hardcoding day count method strings** — use the injected `Map<String, DayCountMethod>` bean to discover available methods dynamically
* {bad} **Calling `setState()` on a loaded `CreditProduct` inside a long-running transaction** — prefer `CreditProductService`'s `getReference()` approach; otherwise you pay the cost of loading `additives` even when they aren't needed
* {bad} **Overriding `ProductWebConfigurationSupport.initDefinitions()` without calling `super.initDefinitions()`** — silently drops the 12 SDK beans; controllers and actions vanish with no error at startup
* {bad} **Returning entity instances from `findAdditiveModelsByProduct(...)`** — templates will trigger lazy loads per row; always return pre-materialized DTOs
* {bad} **Rendering `RateEvaluationException.getMessage()` in the UI** — leaks expression text and internal keys (`rate123456`); surface `getVisibleMessage()` instead
* {bad} **Using native SQL to read additives without re-applying `active = true`** — the `@SQLRestriction` is a Hibernate-layer filter; native queries see soft-deleted rows
* {bad} **Calling `additiveRepository.removeAdditive(productId, additiveId)` without verifying parent state is DRAFT** — the native update has no guard; enforce state in the caller or in an action `when(...)` predicate
* {bad} **Skipping `reloadPageHelper.reload(...)` in parent-child actions** — page stays stale after child mutation; `AddCreditProductAdditiveAction` is the canonical example of why you need it
