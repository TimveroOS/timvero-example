= Covenant Monitoring System

:sourcedir: ../../main/java/com/timvero/loan

The Covenant Monitoring System provides automated control of the financial conditions
of loan agreements through integration with the Metric System and the Feature Store.

== Architecture Overview

The Covenant Monitoring System follows a four-layer architecture:

* **CovenantSpecification** — versioned monitoring rules
* **CovenantExecution** — scheduling and performing checks
* **CovenantResult** — storing check results and managing violations
* **Metric Integration** — calculation of indicators via `MetricService`

[source]
----
Scheduled Task → CovenantSpecification → MetricService → CovenantExecution → CovenantResult
                                              ↓
                                    Feature Store + DataSources
----

**Key Principle:** Covenants always operate through the Metric System.
Indicators are computed via `MetricMapping`, and validation conditions are defined through `Expression`.

=== Core System Components

[cols="1,3",options="header"]
|===
|Component |Description

|**CovenantSpecification**
|A versioned configuration of a monitoring rule. Defines the holder, subject, metrics, and validation conditions.

|**CovenantExecution**
|A record of an executed check. Contains status, a collection of results, and error information.

|**CovenantResult**
|The outcome of a check for a specific subject. Possible states: `CLEAN`, `VIOLATION`, `RESOLVED`, `EXCEPTION`.

|**HasCovenant**
|Marker interface for entities that own covenants (e.g., `ExampleCredit`).

|**HasMetric**
|Marker interface for entities being evaluated (e.g., `Participant`, `Collateral`).

|**CovenantHolderResolver**
|Links a holder with its subjects and defines the logic for resolving the entities to be validated.
|===

=== What the Framework Does

**Automatic Monitoring:**

* **Periodic Execution** — checks are performed on a schedule
* **Metric Calculation** — via Feature Store and DataSources
* **CovenantExecution Creation** — generated for each scheduled check
* **Result Persistence** — stores outcomes for all subjects

**Lifecycle Management:**

* **Producer** — creates tasks with status `CovenantExecution.NEW`
* **Consumer** — executes the checks and saves the results
* **Exception Handling** — automatic processing of metrics in `EXCEPTION` state
* **Versioning** — managed through `lineageId`

=== What is a Covenant?

A **Covenant** is a contractual financial condition that a borrower agrees to maintain throughout the term of a loan.
Covenants are used for risk monitoring and early detection of financial distress.

**Typical Covenant Examples:**

* **Borrower Income** — must not decrease by more than 20% from the baseline value
* **Loan-to-Value (LTV)** — loan amount to collateral value ratio < 80%
* **Debt-to-Income (DTI)** — debt-to-income ratio < 40%
* **Minimum Liquidity** — available funds > €10,000

=== CovenantSpecification — Rule Configuration

`CovenantSpecification` defines **what** to check, **who** to check, and **how often** the checks are performed.

[cols="1,3",options="header"]
|===
|Field |Description

|**holderType**
|The type of entity that owns the covenant (`ExampleCredit`). Defined via `CovenantSpecificationHolderType`.

|**subjectKey**
|Specifies who is being checked: `BORROWER`, `GUARANTOR`, `COLLATERAL`. Must match a key from `CovenantHolderResolver`.

|**metricMapping**
|The `MetricMapping` used to compute the current value of the indicator (`REGULAR` metric).

|**anchoredMetricMapping**
|Optional `MetricMapping` used to obtain the baseline value (`ANCHORED` metric).

|**expression**
|JavaScript/Groovy expression used to evaluate the condition.
Returns a `boolean`: `true` = `CLEAN`, `false` = `VIOLATION`.

|**executionType**
|Execution type. Currently only `SCHEDULED` (runs periodically).

|**periodicity**
|Frequency of execution: `DAYS`, `WEEKS`, `MONTHS`, `YEARS`.

|**numberOfPeriods**
|Number of periods between executions (e.g., `1` for monthly checks).

|**dateTimeExecution**
|Date and time of the first scheduled execution. The Producer calculates subsequent ones based on periodicity.

|**additives**
|List of product additive IDs. The covenant applies only to loans with these products.

|**active**
|Activity flag. Only active specifications are executed.
|===

**Versioning:**

* `CovenantSpecification` extends `HistoryEntity` — versioning is supported
* All versions are linked through a shared `lineageId`
* When a specification is edited, a new version is created with `active=true`
* Previous versions remain in history with `active=false`

=== CovenantExecution — Check Record

`CovenantExecution` is created for each run of a check based on a `CovenantSpecification`.

[cols="1,3",options="header"]
|===
|Field |Description

|**specification**
|Reference to the `CovenantSpecification` being executed.

|**periodIndex**
|The index of the period for scheduled executions. Calculated based on `dateTimeExecution` and `periodicity`.

|**status**
|`NEW` → `EVALUATED` or `DISABLED`.
The Consumer processes `NEW` executions and sets the status to `EVALUATED` upon completion.

|**covenantResults**
|A collection of `CovenantResult` entries for all evaluated subjects.

|**exception**
|`ExceptionEntity` if the check failed (e.g., script error, timeout, etc.).

|**createdAt**
|Timestamp when the execution was created.

|**createdBy**
|User who triggered the execution (for manual runs) or `null` (for scheduled ones).
|===

=== CovenantResult — Check Result

`CovenantResult` stores the outcome of a check for a specific subject within a `CovenantExecution`.

[cols="1,3",options="header"]
|===
|Field |Description

|**execution**
|Reference to the corresponding `CovenantExecution`.

|**specification**
|Reference to the `CovenantSpecification` (denormalized for faster access).

|**ownerId**
|UUID of the holder entity (e.g., `ExampleCredit.id`).

|**subjectId**
|UUID of the evaluated subject (e.g., `Participant.id`).

|**metric**
|Reference to the computed `REGULAR` metric.

|**rawContent**
|The raw result of the expression evaluation (string representation).

|**state**
|`CLEAN`, `VIOLATION`, `RESOLVED`, `EXCEPTION`.

|**createdAt**
|Timestamp of result creation.
|===

**CovenantResult States:**

[.unstyled]
* {ok} **CLEAN** — condition satisfied, no violations
* {bad} **VIOLATION** — condition violated
* {ok} **RESOLVED** — violation manually resolved by a user
* {bad} **EXCEPTION** — error occurred during metric or expression evaluation

=== HasCovenant and HasMetric

Marker interfaces for type-safe linking of entities.

=== Metric Types: REGULAR vs ANCHORED

The system uses two types of metrics for different monitoring purposes.

==== REGULAR Metrics

**Purpose:** Represents the current value of an indicator during each evaluation.

**When calculated:** On every covenant execution via `MetricService.getMetric()`.

**Examples:**

* Current monthly income of the borrower
* Current market value of the collateral
* Current debt-to-income ratio

==== ANCHORED Metrics

**Purpose:** Fixed baseline value used for comparison.

**When set:**

* At loan origination — automatically via `AnchoredMetricService.calculateAnchored()`
* When conditions change — recalculated via `recalculateAnchored()`
* Manually by a user — through `UpdateAnchoredMetricAction`

**Examples:**

* Income at the time of loan origination
* Initial appraised value of the collateral
* DTI ratio at loan approval

=== CovenantHolderResolver — Linking holder ↔ subjects

`CovenantHolderResolver` defines relationships between a holder and its subjects.

== Complete Implementation Example

Below is a complete implementation of the covenant system for a credit product using `ExampleCredit` as the holder.

=== Overview

In this example, we implement borrower income monitoring to verify that income has not dropped by more than 20% from the baseline captured at loan origination.

**Implementation components:**

* **CovenantHolderResolver** — links `ExampleCredit` with `Participant` subjects
* **MetricMapping** — config for computing income via the Feature Store
* **CovenantSpecification** — the income verification rule
* **UI Integration** — tabs to browse results

=== Step 1: CovenantHolderResolver Implementation

Create a resolver that links the credit to the participants being evaluated.

[source,java]
----
@Component
public class CreditCovenantHolderResolver implements CovenantHolderResolver<ExampleCredit> {

    @Autowired
    private ExampleCreditRepository creditRepository;
    @Autowired
    private CovenantSpecificationRepository covenantSpecificationRepository;

    @Override
    public Stream<ExampleCredit> resolveTargets(CovenantSpecification specification) {
        // Find all credits to check by product additive
        return specification.getAdditives().stream()
            .flatMap(a -> creditRepository.getAllByAdditiveId(a));
    }

    @Override
    public List<SubjectRecord<ExampleCredit>> getCovenantSubjects() {
        return List.of(
            // BORROWER subject
            SubjectRecordBuilder.<ExampleCredit>builder()
                .name(ParticipantRole.BORROWER.name())
                .subject(new SubjectSupplierRecord<>(
                    Participant.class,
                    credit -> List.of(credit.getApplication().getBorrowerParticipant())
                ))
                .build(),

            // GUARANTOR subject
            SubjectRecordBuilder.<ExampleCredit>builder()
                .name(ParticipantRole.GUARANTOR.name())
                .subject(new SubjectSupplierRecord<>(
                    Participant.class,
                    credit -> credit.getApplication().getParticipants().stream()
                        .filter(p -> p.getRoles().contains(ParticipantRole.GUARANTOR))
                        .collect(Collectors.toList())
                ))
                .build()
        );
    }

    @Override
    public Collection<CovenantSpecification> getCovenantSpecifications(ExampleCredit target) {
        // Find all active specs for the product additive of this credit
        return covenantSpecificationRepository.findAllByAdditiveIdAndActiveTrue(
            target.getCondition().getSecuredOffer().getOriginalOffer().getProductAdditive().getId()
        );
    }
}
----

**Key Points:**

* **@Component** — Spring automatically discovers the resolver via its generic type
* **resolveTargets** — finds credits by the product additive from the specification
* **getCovenantSubjects** — three subject types: BORROWER, GUARANTOR, COLLATERAL
* **getCovenantSpecifications** — filters specs by the credit’s product additive

// Extract income from the Feature Store
var incomeData = features.get("tink", "monthly_income");

if (incomeData == null) {

=== Step 5: ANCHORED Metric Setup

Set the baseline income value at the time of loan origination.

==== Automatic Setup

When the credit transitions to an active state, invoke:

[source,java]
----
@Component
public class CreditChecker extends EntityChecker<ExampleCredit, UUID> {

    @Autowired
    private AnchoredMetricService anchoredMetricService;

    @Override
    protected void registerListeners(CheckerListenerRegistry<ExampleCredit> registry) {
        registry.entityChange().inserted();
    }

    @Override
    protected boolean isAvailable(ExampleCredit credit) {
        return true;
    }

    @Override
    protected void perform(ExampleCredit credit) {
        TransactionUtils.afterTransaction(() -> {
            anchoredMetricService.calculateAnchored(credit);
        });
    }
}
----

**What happens:**

* Finds all `CovenantSpecifications` with `anchoredMetricMapping != null`
* Resolves subjects via `CovenantHolderResolver`
* Computes metrics via `MetricService`
* Persists them as `MetricType.ANCHORED`

== Expression Writing Guide

An expression defines the covenant condition. It is JavaScript or Groovy code that must return a boolean result.

=== Expression Basics

**Expression requirements:**

* **Returns boolean** — `true` for `CLEAN`, `false` for `VIOLATION`
* **Short and readable** — avoid overly complex logic
* **Null-safe** — always check for `null` before use
* **Deterministic** — same input → same output

**Supported scripting engines:**

[.unstyled]
* **JavaScript** — recommended; fast and simple
* **Groovy** — for advanced logic with Java interop

=== Available Bindings

An expression has access to the following variables:

[cols="1,2,3",options="header"]
|===
|Binding |Type |Description

|`covenantHolder`
|`HasCovenant` (e.g., `ExampleCredit`)
|The entity that owns the covenant. All fields accessible via getters.

|`subject`
|`HasMetric` (e.g., `Participant`)
|The entity being evaluated. Fields accessible via getters.

|`metric`
|`Metric` object
|Metric object with methods `getName()`, `getContent()`, and `getDataType()`.

|`<metricName>`
|Depends on the metric
|Direct access to the metric value. The name matches `MetricMapping.name`.

|`anchored`
|Depends on the metric or `null`
|Value of the ANCHORED metric. `null` if not set.
|===

=== Expression Examples

==== Example 1: Debt-to-Income Check

**Requirement:** DTI < 40%.

[source,javascript]
----
// Get the monthly payment from the credit schedule
var monthlyPayment = covenantHolder.getSchedule().getPaymentAmount().getNumber().doubleValue();

// Get the current monthly income from the metric
var monthlyIncome = metric;

if (monthlyIncome == null || monthlyIncome <= 0) {
    return false;  // No income = VIOLATION
}

// DTI must be < 40%
var dti = monthlyPayment / monthlyIncome;
return dti < 0.4;
----

==== Example 2: Combined Check

**Requirement:** Income remains stable AND DTI is acceptable.

[source,javascript]
----
// Check 1: income has not dropped by more than 20%
var incomeStable = anchored != null && metric >= anchored * 0.8;

// Check 2: DTI < 40%
var monthlyPayment = covenantHolder.getSchedule().getPaymentAmount().getNumber().doubleValue();
var dtiAcceptable = (monthlyPayment / metric) < 0.4;

// Both conditions must pass
return incomeStable && dtiAcceptable;
----

=== Null Safety Patterns

Always check for `null` before using any values.

==== Early Return

[source,javascript]
----
// Check anchored
if (anchored == null) {
    return true;  // Or false, depending on business requirements
}

// Check metric
if (metric == null) {
    return false;  // No data = violation
}

// Main logic
return metric >= anchored * 0.8;
----

=== Performance Considerations

Each expression is executed for every subject during every evaluation.

==== Best Practices

* **Avoid loops** — prefer simple comparisons
* **Minimize method calls** — cache results in local variables
* **Do not perform DB queries** — use only available bindings
* **Keep it simple** — move complex logic to `MetricMapping`

==== Good Example

[source,javascript]
----
// Fast and efficient
return metric >= anchored * 0.8;
----

==== Bad Example

[source,javascript]
----
// Slow — multiple method calls
for (var i = 0; i < subject.getApplication().getParticipants().size(); i++) {
    var p = subject.getApplication().getParticipants().get(i);
    if (p.getStatus().equals(ParticipantStatus.ACTIVE)) {
        // Complex logic inside loop
    }
}
----

=== Best Practices

==== Expression Design

* **Return boolean explicitly** — `return true` or `return false`
* **Use meaningful variable names** — `income`, `baseline`, not `x`, `y`
* **Test with real data** — use the UI tester before saving

=== Anti-Patterns

[.unstyled]
* {bad} **Not checking null** — causes `EXCEPTION` instead of `VIOLATION`
* {bad} **Overly nested logic** — makes expressions unreadable
* {bad} **Iterating over collections** — poor performance impact
* {bad} **DB queries inside expression** — use only `MetricMapping` + `Feature Store`
* {bad} **Mutation inside expression** — expressions must only evaluate, not modify state
* {bad} **Using console.log** — not available in production, only for debugging
* {bad} **Hardcoded entity IDs** — expressions must work for any entity
* {bad} **Ignoring currency** — always validate the currency for `MonetaryAmount`

=== Expression Checklist

Before saving a `CovenantSpecification`, verify the following:

[.unstyled]
* {todo} Expression returns a boolean
* {todo} All null cases are handled
* {todo} Tested with real data via UI
* {todo} Works for all subject types (BORROWER, GUARANTOR, etc.)
* {todo} Handles the case when `anchored = null`
* {todo} No magic numbers — use clear constants
* {todo} Comments added for non-trivial logic
* {todo} Under 20 lines of code

== Lifecycle & Scheduling

The Covenant Monitoring System uses a **producer–consumer** pattern to automatically execute checks on a schedule.

=== Overview

**Execution Flow:**

[source]
----
CovenantSpecification (active=true)
    ↓
Producer Task (scheduled) → creates CovenantExecution.NEW
    ↓
Consumer Task (scheduled) → processes NEW → saves Results
    ↓
CovenantResult (CLEAN / VIOLATION / EXCEPTION)
----

**Key Components:**

* **CovenantExecutionProducerTask** — scheduled task that creates executions
* **CovenantExecutionProducer** — business logic responsible for creation
* **CovenantExecutionConsumer** — scheduled task that processes executions
* **CovenantExecutionService** — performs checks and persists results

=== Producer Task

The Producer creates `CovenantExecution` records with status `NEW` for further processing.

==== Configuration

[source,properties]
----
# application.properties
covenant.execution.producer.initialDelay=PT1M
covenant.execution.producer.fixedDelay=PT5M
----

**Properties:**

[cols="1,3",options="header"]
|===
|Property |Description

|`initialDelay`
|Delay before the first execution after application startup. Format: ISO-8601 Duration (`PT1M` = 1 minute).

|`fixedDelay`
|Interval between the completion of the previous run and the start of the next one. Recommended: `PT5M` (5 minutes).
|===

**Key Points:**

* **Idempotent** — repeated runs do not create duplicates (checked via `exists`)
* **Period-based** — only one execution per period
* **REQUIRES_NEW** — independent transaction for each specification
* **Synchronous load** — `getSync()` ensures consistency

=== Consumer Task

The Consumer processes executions with status `NEW` and saves the results.

==== Configuration

[source,properties]
----
# application.properties
covenant.execution.consumer.initialDelay=PT2M
covenant.execution.consumer.fixedDelay=PT1M
----

**Properties:**

[cols="1,3",options="header"]
|===
|Property |Description

|`initialDelay`
|Delay before the first run. Should be greater than the Producer’s delay to allow time for execution creation.

|`fixedDelay`
|Interval between runs. Recommended: `PT1M` (1 minute) for fast processing.
|===

**Processing Strategy:**

* **One at a time** — processes one execution at a time
* **FIFO order** — `ORDER BY id DESC LIMIT 1` (can be changed to ASC)
* **Separate transaction** — each execution runs in its own transaction
* **Fail-safe** — failure in one execution does not block others

== Best Practices

Guidelines for designing, implementing, and operating the Covenant Monitoring System.

=== CovenantSpecification Design

==== Naming Conventions

* **Descriptive names** — use “Borrower Income Stability > 80%” instead of “Income Check”
* **Include thresholds** — explicitly state numeric thresholds in names
* **Subject prefix** — start with “Borrower”, “Guarantor”, or “Collateral”
* **Version suffix** — add “v2”, “Q1 2025”, etc., for multiple versions

**Examples:**

**Good examples:**

* Borrower Monthly Income > 80% of Baseline
* Guarantor Active Status Check
* Collateral LTV < 80%
* Borrower DTI < 40% (Revised Q1 2025)

**Bad examples:**

[.unstyled]
* {bad} Inc Check
* {bad} GUAR_STS
* {bad} LTV

==== Expression Design

* **Keep it simple** — under 20 lines, one main logical condition
* **Null-safe first** — check for null at the beginning
* **Use constants** — avoid magic numbers
* **Comment thresholds** — explain where threshold values come from
* **Return explicit boolean** — always use `return true/false`
* **Handle missing anchored** — define behavior when the baseline is null
* **Test all edge cases** — null, zero, and negative values

**Good example:**

[source,javascript]
----
// Borrower monthly income must not drop below 80% of baseline
// Threshold agreed with Risk Management on 2025-01-15

if (anchored == null) {
    return true;  // Skip check if baseline not set
}

if (metric == null || metric <= 0) {
    return false;  // No income data = violation
}

var THRESHOLD = 0.8;  // 20% drop allowed
return metric >= anchored * THRESHOLD;
----

**Bad example:**

[source,javascript]
----
// Don’t do this
return metric >= anchored * 0.8;  // What if anchored is null?
----

=== CovenantHolderResolver Implementation

==== Resolver Design

* **Single responsibility** — one resolver per holder type
* **Immutable subjects list** — use `List.of()` for `getCovenantSubjects()`
* **Efficient resolveTargets** — use indexed queries
* **Filter inactive** — exclude cancelled or closed entities
* **Handle lazy loading** — use `JOIN FETCH` when needed

**Good example:**

[source,java]
----
@Override
public Stream<ExampleCredit> resolveTargets(CovenantSpecification specification) {
    // Efficient query with filtering
    return specification.getAdditives().stream()
        .flatMap(additiveId -> creditRepository
            .findByAdditiveIdAndStatusIn(
                additiveId,
                List.of(CreditStatus.ACTIVE, CreditStatus.GRACE_PERIOD)
            )
            .stream()
        );
}
----

**Bad example:**

[source,java]
----
@Override
public Stream<ExampleCredit> resolveTargets(CovenantSpecification specification) {
    // Inefficient — loads all credits
    return creditRepository.findAll().stream()
        .filter(c -> specification.getAdditives().contains(c.getAdditive().getId()));
}
----

==== Subject Extraction

* **Return empty list** — never return null when no subjects exist
* **Filter active only** — exclude deleted or inactive entities
* **Handle relationships** — use proper JPA fetching
* **Cache if possible** — subjects rarely change
* **Log unexpected states** — when subjects are missing but expected

== Anti-Patterns

Common mistakes when working with the Covenant Monitoring System and how to avoid them.

=== Expression Anti-Patterns

==== Not Checking Null Values

**Problem:** The expression throws a `NullPointerException` instead of returning `VIOLATION`.

**Bad example:**

[source,javascript]
----
// Dangerous — if anchored == null, this will throw an exception
return metric >= anchored * 0.8;
----

**Why it's bad:**

[.unstyled]
* {bad} Creates a `CovenantResult` with `state = EXCEPTION`
* {bad} Masks the real violation
* {bad} Makes problem analysis harder
* {bad} Requires manual intervention

**Good example:**

[source,javascript]
----
// Safe — explicit null check
if (anchored == null) {
    return true;  // Or false, depending on business logic
}

if (metric == null) {
    return false;  // No data = violation
}

return metric >= anchored * 0.8;
----

=== CovenantHolderResolver Anti-Patterns

==== Loading All Entities

**Problem:** The resolver loads all entities without any filtering.

**Bad example:**

[source,java]
----
@Override
public Stream<ExampleCredit> resolveTargets(CovenantSpecification specification) {
    // Loads ALL credits from the database
    return creditRepository.findAll().stream()
        .filter(c -> c.getStatus() == CreditStatus.ACTIVE);
}
----

**Why it's bad:**

[.unstyled]
* {bad} Huge memory footprint
* {bad} Very slow on large datasets
* {bad} Execution timeouts
* {bad} Risk of OutOfMemoryError when scaling

**Good example:**

[source,java]
----
@Override
public Stream<ExampleCredit> resolveTargets(CovenantSpecification specification) {
    // Efficient query with filtering at the DB level
    return specification.getAdditives().stream()
        .flatMap(additiveId -> creditRepository
            .findByAdditiveIdAndStatus(additiveId, CreditStatus.ACTIVE)
            .stream()
        );
}
----

==== Using REGULAR Instead of ANCHORED

**Problem:** Attempting to compare two REGULAR metrics instead of REGULAR vs ANCHORED.

**Bad example:**
----
CovenantSpecification:
Metric Mapping: participant_monthly_income (REGULAR)
Anchored Metric Mapping: null
Expression:
// Tries to compare with a previous value
// But there is no anchored metric!
return metric >= previousValue * 0.8;  // previousValue is unknown
----

**Why it's bad:**

[.unstyled]
* {bad} No baseline for comparison
* {bad} Expression has no access to history
* {bad} `anchored = null` → check is skipped

**Good example:**
----
CovenantSpecification:
Metric Mapping: participant_monthly_income (REGULAR)
Anchored Metric Mapping: participant_monthly_income (ANCHORED)
Expression:
if (anchored == null) {
    return true;
}
return metric >= anchored * 0.8;
----

== Metric SDK Internals

This section documents the Pumpkin SDK `com.timvero.loan.metric` package — the 9 classes that
implement the metric evaluation layer sitting between `MetricMapping` (definition) and
`CovenantResult` (outcome). Understanding these internals matters when you need to:

* Debug why a metric returns `EXCEPTION` instead of a value
* Understand what `MetricType.ANCHORED` versus `MetricType.REGULAR` actually stores
* Extend the metric tabs/actions for a new holder or subject class
* Reason about what happens inside `MetricService.getMetric()` during an expression evaluation

NOTE: This is post-approval covenant monitoring infrastructure. For pre-approval decision data
on the same subject, see <<feature-service>> — both layers share the `FeaturedSubject` base
interface but serve different business purposes.

=== Package Layout

The metric module ships in three subpackages, each with a single responsibility:

|===
|Subpackage |Files |Responsibility

|`metric` (root)

=== Metric Entity — Immutable Time-Series Record

`Metric` is an append-only entity. Each call that writes a value inserts a new row rather than
updating an existing one, which is why the repository queries are all "find latest by owner and
mapping lineage" — history is preserved for audit.

[source,java]
----
@Entity
@Table(name = "metric")
@Immutable
public class Metric extends UUIDPersistable {
    @Column(nullable = false) private UUID ownerId;
    @Column(nullable = false) private Instant createdAt;
    @Column(nullable = false) private String name;

    @Column(name = "metric_type", nullable = false)
    @Enumerated(EnumType.STRING)
    private MetricType metricType;   // ANCHORED | REGULAR

    @Column(nullable = false)
    @Enumerated(EnumType.STRING)
    private FeatureDataType dataType; // STRING | INTEGER | DECIMAL | BOOLEAN | EXCEPTION | ...

    @Column(length = 4000) private String rawContent;

    @ManyToOne
    @JoinColumn(name = "mapping_id", nullable = false)
    private MetricMapping mapping;
}
----

Key observations:

* `@Immutable` + no setters — never mutate an existing `Metric`, always create a new one
* `rawContent` is serialized via `FeatureDataType.serialize(content)` and parsed back via
  `getContent()` at read time — the same serialization layer used by `Feature`
* `ownerId` is a raw `UUID` (not a `@ManyToOne`) so the metric table stays polymorphic across
  every `HasMetric` subject class without a discriminator
* `mapping` references a **specific version** of `MetricMapping`, but queries resolve through
  `mapping.lineageId` — so changing the mapping's expression does not orphan historical metrics

=== HasMetric Marker Interface

`HasMetric extends FeaturedSubject`. Implementing it does four things:

1. Registers the class in `MetricService.entityClasses` at startup (via JPA metamodel scan)
2. Makes the class selectable as `MetricMappingEntityType` in the admin UI
3. Makes the class a valid `subject` binding inside metric expressions
4. Lets `HasMetricAnchoredMetricTab<E>` render an "Anchored metric" tab on the detail page

[source,java]
----
public interface HasMetric extends FeaturedSubject {
    default String details() {
        return getPrimaryId();
    }
}
----

Override `details()` to return a human-readable description for admin UI rendering —
defaulting to `getPrimaryId()` gives you a UUID, which is useless in the "Update anchored
metric" dropdown. See `Participant`/other `HasMetric` entities in `timvero-example` for
real examples.

=== MetricService — Expression Evaluation Pipeline

`MetricService` is the entry point for computing any metric. The critical method is
`getMetric(subject, mappingName, metricType, persist)`, and the flow is:

[source,java]
----
// 1. Look up the active MetricMapping by name
MetricMapping mapping = mappingRepository.getMetricMapping(mappingName)
    .orElseThrow(...);

// 2. Validate that subject.getClass() matches mapping.metricMappingEntityType
Class<? extends HasMetric> expectedClass =
    entityClasses.get(mapping.getMetricMappingEntityType());
if (!expectedClass.isAssignableFrom(subject.getClass())) {
    throw new IllegalArgumentException("Mapping X does not support subject of class Y");
}

// 3. Evaluate the expression with bindings: { entity, features }
Map<String, Object> bindings = Map.of(
    "entity", subject,
    "features", new FeatureContainer(subject, featureService, persist)
);
Object content = scriptManager.compile(mapping.getExpression()).eval(bindings);

// 4. Wrap result as Metric with resolved FeatureDataType
FeatureDataType dataType = FeatureDataType.resolveDataType(content);
Metric metric = new Metric(subject.getId(), mapping.getName(),
                            dataType, content, mapping, metricType);

// 5. If persist=true, save ONLY if rawContent/dataType differs from the last row
return persist ? save(metric) : metric;
----

Four behaviours worth memorising:

* {ok} **Any exception during evaluation is caught and stored as `Metric` with
  `FeatureDataType.EXCEPTION`.** The metric row always exists — covenants downstream see
  "exception" rather than "missing". `createExceptionMetric()` handles this branch.
* {ok} **Deduplication.** `save()` compares the new row against the latest row for
  `(ownerId, mapping.lineageId, metricType)` and skips the insert if `rawContent` +
  `dataType` are identical. This prevents the metric table from growing on every no-op recheck.
* {ok} **Class registry is built once at startup.** `init(EntityManager)` walks the JPA
  metamodel, finds every class whose interfaces include `HasMetric`, and maps
  `MetricMappingEntityType(simpleName) → Class`. Add `HasMetric` to a new entity and it
  appears in the admin UI after the next restart — no config entry required.
* {bad} **`FeatureContainer.get()` throws `RuntimeException` if the referenced feature has
  `FeatureDataType.EXCEPTION`.** That becomes the metric's EXCEPTION content — you cannot
  recover a "partial" metric from a partially-failed feature datasource.

=== AnchoredMetricService — Holder-Level Orchestration

`AnchoredMetricService` sits one level above `MetricService` and knows about **covenant
holders**. It iterates through every `CovenantSpecification` on a holder, resolves the subjects
via `CovenantExecutionService`, and calls `MetricService.getMetric(..., ANCHORED, true)` for
each pair.

Three public methods with distinct semantics:

[cols="1,3",options="header"]
|===
|Method |When to call

|`calculateAnchored(holder)`
|Unconditional — ALWAYS writes a new anchored metric (subject to dedup in `save()`).
Called at loan origination / disbursement to capture the baseline.

|`recalculateAnchored(holder)`
|Conditional — only processes subjects where the **latest** ANCHORED metric is currently
`EXCEPTION`. Safe to call on a schedule; will not overwrite a good baseline with a freshly
computed one.

|`recalculateAnchored(holder, subject)`
|Same as above, narrowed to a specific subject — called after manual edits or after a
subject's data becomes available.
|===

All three use `@Transactional(propagation = REQUIRES_NEW)` because anchored recalculation is
a side-effect that should not roll back the surrounding business transaction.

See the production `CreditChecker` (`timvero-example/src/main/java/.../credit/checker`) for
the expected call site: `calculateAnchored(credit)` runs inside the disbursement checker.

=== AbstractUpdateAnchoredMetricAction — Manual Baseline Patch

Anchored values sometimes need human override — an appraisal is disputed, a document changes
retroactively, compliance requires a specific baseline. This action provides a form that
enumerates all covenant subjects on the holder and lets the operator pick a subject + mapping,
enter a typed value, and persist it as a new ANCHORED `Metric`.

[source,java]
----

@Controller
@Order(10000)
public class UpdateAnchoredMetricAction
        extends AbstractUpdateAnchoredMetricAction<ExampleCredit> { }
----

Form model:

* `subjectId` — which `HasMetric` subject the value applies to
* `anchoredMetricId` — which `MetricMapping` (picked from the holder's covenant specs)
* `contentType` — restricted by the base class to `STRING` or `INTEGER`
* `content` — the new value; validated by round-tripping through
  `FeatureDataType.parse(serialize(content))` and rejecting with `error.metric.typeValue.notMatch`
  if serialisation fails

The action deliberately does not trigger a recalculation — it just inserts a new ANCHORED row
and reloads the page. The next covenant execution will read the fresh baseline through
`MetricRepository.findLastByOwnerIdAndMappingIdAndMetricType(...)`.

=== MetricRepository — Latest-Per-Lineage Queries

Every repository query follows the same pattern: **filter by owner and metric type, then take
the row with `MAX(createdAt)` grouped by `mapping.lineageId`**. This is what makes the
append-only design usable — callers always see the current state per mapping lineage, while
audit keeps every historical write.

Four queries exist:

[cols="1,3",options="header"]
|===
|Method |Used by

|`findLastByOwnerIdAndMappingIdAndMetricType`
|`MetricService.readMetric` / `save` — latest value for a single mapping
(resolved via `lineageId`, not `id`).

|`findLatestMetricsByOwnerId(owner, type)`
|`HasMetricAnchoredMetricTab` / `HasCovenantAnchoredMetricTab` — list of latest values across
all mappings for a subject.

|`existsLatestMetricsByOwnerIdAndDataType(owner, type, dataType)`
|`AnchoredMetricService.hasAnchoredException` — "does any latest ANCHORED metric have
`EXCEPTION` datatype for this subject?"

|`existsLatestMetricsByOwnerId(owner, type)`
|`HasMetricAnchoredMetricTab.isVisible()` — hides the tab if nothing has been anchored yet.
|===

Custom reporting queries should follow the same lineage-aware pattern; a naive
`ORDER BY createdAt DESC LIMIT 1` will pick a historical row if you upgrade a mapping and
don't re-anchor.

=== MetricMapping Entity

[source,java]
----
@Entity
@Table(name = "metric_mapping")
@Cache(usage = CacheConcurrencyStrategy.READ_WRITE)
public class MetricMapping extends HistoryEntity implements NamedEntity {
    @Column(nullable = false)
    private String name;

    @Embedded
    private Expression expression;

    @Column(name = "metric_entity_type", nullable = false)
    @Type(value = NamedParameterType.class)
    private MetricMappingEntityType metricMappingEntityType;
}
----

Three things drive everything downstream:

* **`extends HistoryEntity`** — each edit is a new row sharing a `lineageId` with the previous
  version; only rows with `active = true` are considered by `MetricMappingRepository.getMetricMapping(name)`.
  This is also why `MetricService.save()` dedupes by `mapping.lineageId` rather than `mapping.id`.
* **`@Embedded Expression expression`** — the expression is stored with its engine name
  (JavaScript by default), so the same mapping can reference a different engine across versions.
* **`@Type(NamedParameterType.class) MetricMappingEntityType`** — column-mapped to a plain
  string column (`metric_entity_type`) but type-safe at the Java level. The string is the
  `simpleName` of the target `HasMetric` class, which is why renaming a `HasMetric` entity class
  breaks every mapping pointing at it. Never refactor an entity class name without a data migration.

The 2nd-level cache (`CacheConcurrencyStrategy.READ_WRITE`) matters: `MetricService.getMetric()`
is called inside tight covenant loops; without the L2 cache each expression evaluation would
hit the DB for the `MetricMapping` row.

=== MetricMappingEntityType — Why Not Class?

Admins pick a `MetricMappingEntityType` from a dropdown; the DB stores it as a string. The
SDK chose `NamedParameter` over `Class<?>` for three reasons:

* {ok} Mappings survive when you rename-then-map the class (via custom migration)
* {ok} Mappings are portable across packages/modules — no FQN lock-in
* {ok} The admin UI can show user-friendly names in the dropdown

The bridge between string and class lives in `MetricService.entityClasses`, built at startup
from the JPA metamodel. If a mapping references an entity type that no longer implements
`HasMetric`, `MetricService.validateMapping()` throws `IllegalArgumentException` —
stored as `FeatureDataType.EXCEPTION` content, visible in the admin tab.

=== MetricMappingRepository Queries

Three queries, all lineage-aware:

[source,java]
----
// Active mapping resolved by name — used by MetricService.getMetric(subject, name, ...)
Optional<MetricMapping> getMetricMapping(String name);

// All active mappings for a given entity type — used by MetricService.getAllActive(ownerClass)

The `findLatestActiveByLineageId` query is the reason admins can edit a mapping mid-cycle

metrics write against version N+1, and the history tab shows both.

=== Form Pipeline — MetricMappingFormService

`MetricMappingFormService extends HistoryEntityFormService` inherits the "save = new version"
behaviour. The service adds three responsibilities:

[source,java]
----
@Override
protected void assembleEditModel(MetricMapping mapping, MetricMappingForm form,
                                  Map<String, Object> model) {
    if (form.getExpression() == null) {
        form.setExpression(new ExpressionForm());
        form.getExpression().setEngineName(JSScriptEngine.NAME);   // 1. default engine
    }
    model.put("metricMappingEntityTypes", mappingService.getResultTypes());
    model.put("engineNames", scriptManager.getEngineNames());
    model.put("entityTypes", metricService.getEntityClasses()...); // 2. populate dropdowns
}

@Override
public UUID save(UUID id, MetricMappingForm form) throws ValidationException {
    if (id != null) {
        MetricMapping existing = repository.getReferenceById(id);
        if (!existing.getMetricMappingEntityType().equals(form.getMetricMappingEntityType())) {
            ValidationException.binding()
                .rejectValue("metricMappingEntityType", "metric-mapping.unmodifiableField")
                .throwIfNotEmpty();                                 // 3. freeze entity type
        }
    }
    return super.save(id, form);
}
----

Locking `metricMappingEntityType` after creation is intentional — letting an admin change the
target entity type mid-lifecycle would orphan every historical `Metric` whose `ownerId` points
at the old class.

The `MetricMappingFormMapper` is a standard MapStruct interface; the only interesting mapping
is `@Mapping(target = "lineageId", ignore = true)` — the `lineageId` is managed by
`HistoryEntity`, not the form.

=== Controller — List + Expression Tester

`MetricMappingController` registers itself as a setting menu item:

[source,java]
----
@MenuItem(order = 900_620, name = "metricMapping", group = "setting")
@RequestMapping(MetricMappingController.PATH)  // "/metric-mapping"
----

Two JSON endpoints back the expression tester UI:

* `POST /metric-mapping/evaluate-expression` — runs
  `MetricMappingService.testMetric(expression, type, entityId)` which builds an in-memory
  `MetricMapping("", expression, type)` and calls `metricService.getMetric(subject, mapping)`
  with `persist = false`. The expression is never saved — `name = ""` — and dedup logic does not
  apply because the service does not hit `save()`.
* `GET /metric-mapping/entities?search=...&metricMappingEntityType=...` —
  `ValidationEntitySearchService.search()` against the picked `HasMetric` class, returning a
  `Map<id, label>` that the UI renders as a searchable dropdown.

`@ExceptionHandler(ScriptException.class)` returns 412 Precondition Failed with the localised
message so the tester UI can render a clean error. If you extend the controller with more
endpoints, follow the same pattern rather than letting `ScriptException` bubble up as 500.

=== Filter — Hide Inactive by Default

`MetricMappingFilter extends ListFilter` with a single `@Field(restriction = EQ) boolean active = true`.
The list page shows only active mappings by default; ticking the checkbox reveals archived ones.
`MetricMappingFilterService` is empty — it's just a typed instantiation of
`FilterListService<MetricMapping, MetricMappingFilter>`.

=== Tab — Version History

`MetricMappingDetailsTab` is the default detail tab with `@Order(1000)`. It always renders
(`isVisible() == true`), and exposes four model attributes to the Thymeleaf template:

* `engineNames` — from `ScriptManager.getEngineNames()`, so the engine dropdown is always fresh
* `metric` — the current mapping version
* `history` — all versions sharing the lineage, sorted by `createdAt DESC`
  (via `MetricMappingFormService.getMetricVersions(mapping)`)
* `isNoneActiveVersionExist` — true if every version has `active = false`; the template uses
  this to prompt the operator to activate a version

=== Export / Import Round-Trip

Three classes form a round-trippable catalog export:

* `MetricMappingExportData` — DTO with `name`, `ExpressionForm expression`, and
  `metricMappingEntityType` as a plain `String` (no `@Type` on a DTO). Implements `equals`/`hashCode`
  so the diff tool can detect identical mappings across environments.
* `MetricMappingToDtoMapper` — MapStruct mapper with `@Mapper(uses = NamedParameterMapper.class)`,
  which handles the `NamedParameter` ↔ `String` conversion automatically.
  `lineageId` is ignored on import — new environments get new lineages.
* `MetricMappingExporter extends AbstractHistoryEntityExporter` with `@Order(7400)` and
  `getVersion() = "1"`. The order must place this after any entity class containing `HasMetric`
  is imported, otherwise the string `metric_entity_type` will not resolve at runtime.

When bumping the version, also version the downstream covenant specs — a mapping rename that
leaves `CovenantSpecification.metricMapping` referencing a stale lineage is silent until the
next execution.

=== How Metrics Flow From Credit to Covenant Check

Tying the two packages together, the end-to-end happy path for an ANCHORED-backed covenant is:

[source]
----
CreditChecker (origination)
    │
    ▼
AnchoredMetricService.calculateAnchored(credit)
    │
    ├─ CovenantExecutionService.getSpecifications(credit)       — specs on holder
    │
    ├─ for each spec with anchoredMetricMapping != null:
    │     │
    │     ├─ CovenantExecutionService.getSubjects(credit, spec) — via CovenantHolderResolver
    │     │
    │     └─ for each subject:
    │           │
    │           └─ MetricService.getMetric(subject,
    │                    spec.getAnchoredMetricMapping().getName(),
    │                    MetricType.ANCHORED, persist=true)
    │                 │
    │                 ├─ MetricMappingRepository.getMetricMapping(name)
    │                 ├─ validateMapping(subject, mapping)
    │                 ├─ evaluateMetricExpression(subject, mapping, persist=true)
    │                 │      │
    │                 │      └─ ScriptManager.compile(expression)
    │                 │          .eval({ entity, features(FeatureContainer) })
    │                 │
    │                 └─ save(metric) — dedup by (ownerId, lineageId, metricType)
    ▼
Scheduled consumer (see Lifecycle & Scheduling)
    │
    └─ For each CovenantSpecification on each active holder:
         │
         ├─ Compute REGULAR metric via MetricService.getMetric(..., REGULAR, true)
         ├─ Read ANCHORED metric via MetricRepository.findLastByOwnerIdAndMappingIdAndMetricType
         └─ Evaluate covenant expression with { metric, anchored } bindings
               → CovenantResult.state ∈ { CLEAN | VIOLATION | RESOLVED | EXCEPTION }
----

The same `MetricService.getMetric(...)` signature is used for both ANCHORED (at origination)
and REGULAR (at every scheduled check) evaluations — the only difference is the
`MetricType` argument. This is why the dedup logic in `save()` is keyed on
`(ownerId, mapping.lineageId, metricType)` — a REGULAR metric never overwrites an ANCHORED one.

=== Additional Anti-Patterns — Metric Internals

==== Mutating Metric Fields

**Problem:** Calling a "setter" on an existing `Metric` to change its value.

[.unstyled]
* {bad} `Metric` is `@Immutable` — there are no setters
* {bad} Even if you bypass Hibernate, you destroy the audit trail
* {bad} Covenant executions may already reference this row

**Fix:** Always create a new `Metric(...)` and persist it through `MetricService.saveMetric()`.

==== Renaming a HasMetric Entity Class

**Problem:** `@Entity` class renamed from `Participant` to `LoanParticipant` without a migration.

[.unstyled]
* {bad} Every `MetricMapping` stores the old `simpleName` in `metric_entity_type`
* {bad} `MetricService.validateMapping()` throws on the next execution
* {bad} Metrics stored as `FeatureDataType.EXCEPTION` for all affected subjects

**Fix:** Either keep the class name or write a Flyway migration that rewrites
`metric_mapping.metric_entity_type` in-place.

==== Skipping the `details()` Override

**Problem:** A `HasMetric` entity uses the default `details()` which returns `getPrimaryId()`.

[.unstyled]
* {bad} "Update anchored metric" dropdown shows UUIDs instead of names
* {bad} Audit logs are hard to correlate with business entities
* {bad} `HasCovenantAnchoredMetricTab` rows are unreadable

**Fix:** Override `details()` to return a business-meaningful label
(e.g. `name + " (" + id + ")"` for `Participant`).

==== Using `evaluate-expression` as a Computation Cache

**Problem:** Client code calls `POST /metric-mapping/evaluate-expression` to get "live" metric
values instead of writing an ANCHORED mapping and calling `MetricService.getMetric(...)`.

[.unstyled]
* {bad} The tester endpoint runs with `persist = false` — no Metric row is written
* {bad} Results are not audited, not cached, not visible in tabs
* {bad} Every call recompiles the expression — wasteful under load

**Fix:** Use the tester endpoint strictly for interactive validation. For recurring
computation, define a real `MetricMapping` and call `MetricService` from application code.
